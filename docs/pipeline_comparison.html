Why Three?

BrokerPriceMonitor Original implementation. Watches conditional orders (BTO triggers). Still active — routes.py imports it for order cancellations. Has the simplest architecture but weakest protection.
StreamingPriceMonitor Replacement for BrokerPriceMonitor. Stream-first with frozen detection and cross-broker probing. Primary path for new conditional orders via the router. Coexists with the old monitor.
RiskManager Completely separate pipeline for open position SL/PT/trailing evaluation. REST-snapshot architecture with streaming overlays. Most sophisticated — has sanity guards, REST repair, staleness gates. Does not share code with either conditional order monitor.
Key Problem: Every threshold, every fallback chain, every symbol normalization rule must be maintained in three places. They have already diverged — frozen detection is 3s in one place, 3s/15s in another; freshness checks exist in the risk engine but not in either conditional order monitor; cross-broker REST chains use different broker orders.

FCHL Incident: Which Pipeline Failed?

The Risk Engine evaluated the SL exit — and it triggered late at -16.83%

Root cause chain through the Risk Engine pipeline:

No hub streaming FCHL
Webull MQTT silently ignored this penny stock. _update_prices_from_hub overlay had nothing to overlay.
_request_cross_broker_subscriptions skipped
Webull is_streaming() = True globally → not flagged as "non-streaming broker" → Schwab subscription never requested
No per-symbol streaming tracking
_dirty_symbols cleared each cycle → system doesn't know FCHL never received a streaming tick
REST polling at 3-5s intervals
Global REST refresh: $1.005 at 10:41:47 → next poll $0.889 at 10:41:51 (4s gap)
Stuck detector NOT triggered
Price was changing every REST poll → tracker['last_changed'] reset → stuck_seconds never reached 3s
SL gap-through: -10% → -16.83%
Flash crash happened between two REST polls. No streaming to catch it.
