Executive Summary

What SnapTrade is, what the risk engine is, and why one cannot stand in for the other.

Per-channel risk fields
42
Validated registry in RISK_SETTINGS_VALIDATION_REFERENCE.md (rows 1–48)
Channel-tab control fields
20
Sizing, Risk Controls, Order Types, Conditional
SnapTrade-mappable
7 / 62
Order type, TIF, limit price, qty, side, account routing, partial-fill status

Core message. SnapTrade is an order router: a unified REST API that translates one order intent into the target broker's order intent. It has no concept of profit-target ladders, no EMA candle builder, no millisecond-cadence position monitor, no per-channel state machine, no signal-update automation, and no ticker filter. Those live entirely in src/risk/position_monitor.py, src/risk/risk_types.py, and the channel/execution surfaces of the GUI.

Practical takeaway. If SnapTrade replaces broker SDKs, the Risk Engine still owns 100% of the trade lifecycle decisions — it would simply emit normalized SnapTrade order requests at the moments it currently emits broker-specific ones. The risk engine is not a "feature you can replace"; it is the product.

Targets & Stop Loss

Trading → Execution → Risk Management → Targets & SL tab. This is the heart of the product.

Profit Target Ladder (P1–P4)

FieldDefaultBehaviorSnapTrade?
profit_target_1_pct … profit_target_4_pct10 / 20 / 30 / 40Four cumulative profit thresholds. Engine arms a partial-sell when any target is crossed and not yet filled.No
profit_target_qty_1..4AutoOptional override: exact contracts to trim at each target. Engine falls back to even split when null.No
profit_target_trim_pct_1..4AutoOptional override: % of remaining position to trim at each target. Custom qty wins if both are set.No

Why SnapTrade can't do this. A "PT ladder" is a stateful contract: as each target fills, the engine recomputes remaining qty, may re-issue the next limit, may cancel the prior on a reversal. SnapTrade exposes single-leg orders only; a ladder of four would need four cancel/replace cycles plus client-side bookkeeping — which is exactly what the Risk Engine already does.

Exit Strategy Mode

Signal

Risk engine is silent; only trader-issued STC signals close positions.

Risk

Engine ignores trader STC; only PT/SL/trailing exits.

Hybrid

First trigger wins — whichever fires first (signal or risk) closes the position.

SnapTrade has no concept of "trader-signal exits" — exits would be reduced to whatever order you happen to place. The mode itself is a routing/policy decision in the engine.

Advanced Risk

Trading → Execution → Risk Management → Advanced tab. The differentiated, alpha-bearing controls.

Dynamic Stop-Loss Escalation

  • enable_dynamic_sl — turns the escalator on.
  • dynamic_sl_profile — conservative (PT1: BE, PT2: +3%, PT3: +8%, PT4: +15%) · standard (BE / +5 / +10 / +17) · aggressive (-2 / BE / +8 / +15).
  • escalation_only_mode — targets only ratchet the stop; never partial-sell.

Each PT fill rewrites the stop; engine must be live in-process to do this. SnapTrade has zero hooks for it.

Max Profit Giveback Guard

  • enable_giveback_guard — switch.
  • giveback_allowed_pct — close if peak P/L drops by > X% (5–80, default 30).
  • Activates only after PT2 or trailing-activation has been hit.

Requires a peak-P/L watermark per position — purely engine state.

SnapTrade gap. SnapTrade is a unified order API — it doesn't ship market data, doesn't compute indicators, and doesn't run a per-position scheduler. Ripping the EMA engine out and trying to recreate it client-side just to call SnapTrade for the close order would be strictly more code, not less.

Ticker Filter

Per-channel allow/block list applied to the signal's underlying symbol.

Modes

  • off — no filtering.
  • allow — only signals matching the list trade.
  • block — signals matching the list are skipped.

For options, the underlying is matched (e.g., SPY 450C matches SPY). Case-insensitive comma-separated list.

SnapTrade equivalent

None. SnapTrade ingests orders, not signals. The filter is a routing decision before any order is constructed.

What SnapTrade Actually Provides

A fair accounting of the surface area we'd actually be buying.

SnapTrade strengths (the genuine value)

  • One unified POST /trade/place endpoint across ~14 brokers.
  • Order types: Market, Limit, Stop, StopLimit.
  • TIF: DAY, GTC (broker-dependent).
  • Account & positions normalization (cash, equity, BP).
  • Order status & partial-fill webhook (PARTIALLY_FILLED, FILLED, CANCELLED).
  • OAuth-based broker linking → no password storage.
  • Replaces selfbot / scraping flows for Webull, Robinhood, Trading212 (subject to broker availability).

SnapTrade gaps vs the Risk Engine

  • No PT ladder, no per-target qty/% override, no trim presets.
  • No native trailing, no early-trailing breakeven ratchet.
  • No DYN-SL escalation profiles, no escalation-only mode.
  • No giveback guard / peak-P/L watermark.
  • No EMA engine (no candles, no indicators, no per-position scheduler).
  • No exit-strategy modes (signal / risk / hybrid).
  • No signal-update automation (it doesn't see Discord/Telegram).
  • No NDX→QQQ delta-aware remap.
  • No slippage / limit-cap / order-chase loops.
  • No conditional triggers tied to upstream messages.
  • No ticker filter (allow/block by underlying).
  • No bracket primitive — PT and SL legs must be placed separately.
  • No "force my size" / sizing modes / paper-track sizing split.
  • No Quick Presets (Default / Swing / Momentum / Trend).

Recommendation

How to think about adopting SnapTrade without losing the product.

Keep

Every red-box component above. Profit-target ladder, DYN-SL escalator, EMA engine, giveback guard, early trailing, exit-strategy modes, signal-update automation, NDX→QQQ remap, slippage / chase / conditional / breakout loops, ticker filter, presets.

Wrap

Replace direct broker SDK calls inside the Order Emitter with a thin SnapTrade adapter for the brokers SnapTrade actually supports better than the current code paths (Webull, Robinhood, T212 selfbot). Keep IBKR/Schwab/Alpaca/Tasty native where SnapTrade is more restrictive.

Defer

Conditional / bracket / TIF behaviors should remain engine-side until SnapTrade exposes a unified bracket primitive. Today, every "bracket" through SnapTrade would still be three separate calls and three reconciliations — exactly what we already do natively.

Net effect of a SnapTrade-only stack. You delete ~12% of the routing code (broker-specific order placement) and lose ~88% of the product (everything that decides when, how much, and at what price). The right move is the inverse: keep the engine, treat SnapTrade as one of several optional output adapters.
