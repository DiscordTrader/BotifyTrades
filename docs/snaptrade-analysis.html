00 · OVERVIEW

Executive summary

The one-page version, for the read-only audience.

The question

BotifyTrades runs ~10,700 lines of broker-specific code across 7 modules, and is visibly straining under per-broker quirks (Webull self-bot, Schwab token storms, Robinhood serialization, IBKR Gateway). SnapTrade promises to replace most of this with one API that already supports 20+ retail brokers.

Should we throw away most of the broker layer and adopt SnapTrade — or keep what we have? This document answers that.

The answer

Pure-SnapTrade is the wrong call. SnapTrade has no real-time market-data WebSocket, and adds 200–1500 ms of round-trip latency. Both are deal-breakers for SPX 0DTE-class signals where the live-tick risk engine is the product.

The right call is hybrid: keep Schwab + Webull native, and add SnapTrade as a parallel adapter behind the existing BrokerInterface. We instantly broaden coverage to Fidelity, Vanguard, Tradier, Public, Wealthsimple, Questrade, and others — without touching the hot path.

Three things to remember

1. SnapTrade is fantastic for breadth and onboarding. 2. SnapTrade is bad for latency-critical paths and live ticks. 3. The two-path hybrid is the only design that captures the breadth without giving up the latency edge.

01 · LANDSCAPE

What SnapTrade is

A single API that abstracts away the auth, order, and reflection differences across 20+ retail brokerages — the Plaid for retail trading.

SnapTrade (operated by Passiv) provides a hosted aggregator API plus first-party SDKs for Python, TypeScript, Java, Ruby, C#, and Go. End-users authorize their brokerage through a SnapTrade-hosted Connection Portal; from then on, every call is keyed by userId + userSecret + accountId and your application never handles the user's broker credentials directly.

  • Order types Market · Limit · Stop · StopLimit
  • Time-in-force Day · GTC · FOK · IOC
  • Notional / fractional Yes (where broker permits)
  • Confirmation wait_to_confirm: true
  • Status enum 14 normalized states

Pricing

  • Free tier 5 concurrent connections
  • Pay-as-you-go $2 / user / month
  • Default rate-limit 250 req / min / partner
  • Enterprise Custom volume + Slack support

A user with N broker connections is still billed once (per user, not per connection).

Important nuance

Account data is cached server-side (typically minutes-fresh). For fresh state you call refreshBrokerageAuthorization. There is no native real-time market-data WebSocket — read endpoints are REST + cache + webhooks. This is the single biggest functional gap vs the current setup.

Code surface — visualized

Pain points the codebase visibly bears the cost of

Partial-fill / order verification

Today's get_origin_status_after_stc patch and the SPX 7130C zombie-position incident both stem from optimistic DB writes that don't wait for broker fill confirmations.

Symbol normalization

Index option mapping (SPX↔SPXW, NDX, RUT) is special-cased in position_monitor.py and an INDEX_ALIASES_IN table in selfbot_webull.py.

Token-refresh storms

Schwab uses _token_refresh_lock + exponential backoff to avoid hammering its auth server when tokens expire mid-trade.

Per-broker quirks

Schwab 120/min budget, Webull "system busy" retries, Robinhood serialization to avoid account locks, Trading 212 timeout logging.

Self-bot fragility

Webull's selfbot_webull.py is 21,548 lines and must be kept warm with a periodic trade-token refresh loop.

No unified status enum

Every broker reports order state differently; reconciliation is ad-hoc and partial-fill semantics drift between modules.

04 · PERFORMANCE

Latency & throughput

Where the trade-off lives. The risk engine fires every ~180 s and reacts to live ticks; here's what gets between us and the broker.

Why native is faster

Webull and Schwab open a persistent WebSocket. Orders are issued on already-warm TLS sessions. Schwab's REST path uses a single TLS round-trip plus the broker's own ack.

Why SnapTrade is slower

You hit SnapTrade over HTTPS, SnapTrade re-issues the order to the broker over their session, then proxies the response back. The 2-step impact → place flow doubles the round-trip.

Throughput

Default partner rate-limit is 250 req/min across all your users. That is fine for portfolio dashboards; for a 1k-user signal-following load with bursty closes it needs the Enterprise tier.

Latency is the headline. Everything else is negotiable; latency is not — the risk engine eats latency for breakfast.

08 · IMPLEMENTATION

What the SnapTrade adapter looks like

A sketch of SnapTradeBroker(BrokerInterface) that plugs into the existing BrokerFactory with no other changes.

Onboarding flow

# 1. provision a SnapTrade-side user, store the secret
from snaptrade_client import SnapTrade

st = SnapTrade(client_id=ST_CLIENT, consumer_key=ST_KEY)

resp = st.authentication.register_snap_trade_user(
    user_id=str(local_user_id),
)
db.save_st_secret(local_user_id, resp.body["userSecret"])

# 2. send the user to the hosted Connection Portal
login = st.authentication.login_snap_trade_user(
    user_id=str(local_user_id),
    user_secret=secret,
    broker="SCHWAB",          # or "ROBINHOOD", "FIDELITY"...
    immediate_redirect=True,
    custom_redirect=f"{APP_URL}/oauth/return",
)
return redirect(login.body["redirectURI"])

Place an option STC and wait for confirmation

def sell_to_close(account_id, option_symbol, qty, limit):
    impact = st.trading.get_order_impact(
        account_id=account_id, user_id=u, user_secret=s,
        body={
            "action": "SELL_TO_CLOSE",
            "option_symbol": option_symbol,
            "order_type": "Limit",
            "price": limit,
            "units": qty,
            "time_in_force": "Day",
        })
    confirmed = st.trading.place_order(
        trade_id=impact.body["trade"]["id"],
        user_id=u, user_secret=s,
        wait_to_confirm=True,    # blocks until terminal
    )
    return confirmed.body["status"], confirmed.body["filled_quantity"]

Adapter shape

class SnapTradeBroker(BrokerInterface):
    def __init__(self, user_id, user_secret, account_id):
        self.client = SnapTrade(client_id=ST_CLIENT, consumer_key=ST_KEY)
        self.uid, self.secret, self.acct = user_id, user_secret, account_id

    def place_option(self, symbol, side, qty, limit, tif="Day"): ...
    def cancel_order(self, broker_order_id): ...
    def get_positions(self): ...
    def get_orders(self, since=None): ...
    def on_webhook(self, event):       # register at /webhooks/snaptrade
        if event["type"] == "ORDER_UPDATED":
            db.upsert_fill(event["data"])

All five methods exist already on the existing native brokers, so the dispatcher in broker_interface.py doesn't change. Webhook intake is the only new HTTP route.

09 · ROADMAP

Migration roadmap (Hybrid path)

A 6-week plan to ship Option B without disrupting the live trading loop.

Week 1–2 · Foundation

Spin up a SnapTrade developer account. Build a 200-LOC POC against the Python SDK. Measure end-to-end latency on the production VPS for both equity and options.

Week 3–4 · Adapter

10 · DECISION

Recommendation

Adopt the Hybrid (Option B) path.

For BotifyTrades' use case — option-signal following with sub-second risk management on hot tickers like SPX 0DTE — pure SnapTrade is the wrong call. The latency edge and live-tick risk engine are the product.

Instead: keep Schwab + Webull (and maybe Alpaca) native for hot-path execution and streaming, and add SnapTrade as a parallel adapter behind the existing BrokerInterface. Instantly broaden coverage to Fidelity, Vanguard, Tradier, Questrade, Public, Wealthsimple, etc. for users whose brokerage we don't currently support and whose orders aren't latency-sensitive.

Concrete next steps

  1. Draft the SnapTradeBroker(BrokerInterface) adapter using the Python SDK so it plugs into the existing BrokerFactory.
  2. Build a 200-LOC POC: register a test SnapTrade user, open the Connection Portal, list accounts, pull positions — measure latency on the VPS empirically before committing.
  3. Pull listAllBrokerages to confirm exactly which of the current 7 brokers SnapTrade supports natively today, and which third-party brokers to surface to users first.
  4. In parallel: fix the optimistic-fill DB write (the SPX 7130C zombie root cause) — that bug exists today on the native path and SnapTrade adoption shouldn't be a prerequisite for fixing it.
