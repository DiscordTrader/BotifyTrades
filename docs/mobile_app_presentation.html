01 — Core Architecture

Hub-and-Spoke Relay

botifytrades.com acts as a lightweight relay — it authenticates users and bridges WebSocket messages between the desktop bot and mobile app. All trading execution stays 100% local on the user's machine.

Privacy First

Broker credentials, API keys, and Discord tokens never leave the desktop. The server is a dumb pipe — it relays encrypted WebSocket messages without reading trade data.

⚡

Zero Infrastructure

Works behind NAT, firewalls, and routers. The desktop bot connects outbound to the relay — no port forwarding, no dynamic DNS, no VPN. Just works.

💰

Not SaaS

The server costs ~$15/month to run. No per-user compute, no broker API proxying, no trade execution in the cloud. All heavy lifting stays on the user's machine.

Renders positions list with live P&L — continues receiving updates via WebSocket every 5 seconds

🚨 Scenario 2: Stop Loss Triggered → Push Notification

Push: "SL Triggered: AAPL 185C closed at -$120.00"

Phone buzzes with push notification. User taps → opens alert detail in app.

🚫 Scenario 3: Close Position from Phone

User sees a losing position on their phone and decides to close it remotely (requires permission enabled on desktop).
1
Mobile App
User taps "Close" on SPY 450P position → confirmation dialog → confirmed
{"type": "command", "action": "close_position", "symbol": "SPY 450P", "broker": "IBKR"}
2
Relay Server
Validates JWT, checks rate limit, forwards command to user's paired bot
3
Desktop Bot
Checks permissions — "allow_close" must be enabled. If allowed, executes STC on IBKR locally.
4
Desktop Bot
Sends acknowledgement + trade result back through relay
{"type": "command_ack", "success": true} + {"type": "trade", "action": "STC", "pnl": -23.00}
5
Mobile App
Shows success toast. Position disappears from list. Trade logged in history.
05 — WebSocket Protocol

Message Format

All communication uses JSON over WebSocket. The bot pushes status upstream; the server relays commands downstream.

▲ Bot → Server (Upstream)

heartbeat — every 30s to keep connection alive
"ts": 1716000000
status — every 15s, bot health snapshot
"brokers": ["SCHWAB", "IBKR"], "positions": 3, "pnl": 345.80
positions — every 5s (when open positions exist)
[{"symbol": "AAPL", "pnl": 67.0, "broker": "SCHWAB"}]
alert — instant, triggers push notification
"level": "high", "msg": "SL triggered AAPL -$120"
trade — instant, every fill
"action": "BTO", "symbol": "SPY 450C", "price": 3.50

▼ Server → Bot (Commands)

command: "pause_trading"
Pauses all signal processing and order execution
command: "resume_trading"
Resumes normal trading operations
command: "close_position"
"symbol": "AAPL", "broker": "SCHWAB"
command: "close_all"
Emergency close all positions across all brokers
command: "cancel_conditional"
"order_id": 55
command: "request_status"
Force immediate status update from bot

07 — Security Model

Privacy by Architecture

The relay server is designed to know as little as possible. Broker credentials never leave the desktop. Optional end-to-end encryption makes the server completely blind to message content.

● What the Server Knows

  • ✓ User email + hashed password
  • ✓ Bot online/offline status
  • ✓ Bot version number
  • ✓ Mobile device push tokens
  • ✓ Relay messages (transient, not stored)

● What the Server NEVER Sees

  • ✗ Broker credentials or API keys
  • ✗ Discord token
  • ✗ Account numbers or balances
  • ✗ Full trade history (permanent)
  • ✗ Position sizes or portfolio value

Optional End-to-End Encryption

For maximum privacy, bot and mobile share an AES-256-GCM key during pairing. The server relays encrypted blobs it cannot read.
