⚠ Verdict: Ambitious but Feasible — with Critical Adjustments

The architecture document is exceptionally comprehensive (50 sections, 1,500+ lines) and demonstrates deep understanding of the trading domain. However, it over-architects for Day 1. With 20 microservices, 40+ database tables, and 50+ event topics, it describes a Year 3 target state rather than an MVP. The biggest risks are SnapTrade capability gaps (OCO/bracket support, rate limits at scale), latency degradation from desktop <100ms to SaaS queue-based execution, and regulatory exposure from executing trades on behalf of users. The document is a strong north-star reference — the execution plan needs right-sizing.

Critical Gap Analysis

14 gaps identified across 4 severity levels. Critical and High gaps must be resolved before development begins.

P0
1. SnapTrade Capability Assumptions Are Unvalidated
The entire execution layer assumes SnapTrade supports: OCO/bracket orders, streaming price data, trailing stops, partial cancel/replace, and option-specific order types. None of these are confirmed. SnapTrade is primarily a connection aggregator — it proxies to broker APIs but does not guarantee feature parity with direct API access.
Blocker Execution Risk
Remedy: Before any development, conduct a SnapTrade API capability audit. Test OCO placement on Schwab via SnapTrade. Test option orders. Measure rate limits. Build a compatibility matrix. If SnapTrade can't do OCO — the entire bracket order architecture needs a software-only fallback from Day 1.
P0
2. Regulatory/Legal Risk Is Under-Addressed
Executing trades in user brokerage accounts based on third-party signals may constitute "investment advice" or "discretionary management" under SEC/FINRA rules. The architecture treats compliance as a disclaimer — it may actually require registration as an RIA (Registered Investment Adviser), broker-dealer exemption analysis, or falling under the "publisher's exclusion." This is not a technical problem but it can shut down the platform.
Blocker Legal Risk
Remedy: Engage a fintech compliance attorney before building. Specifically evaluate: (a) Does auto-executing third-party signals for users require RIA registration? (b) Does the "self-directed" framing hold if the platform auto-executes? (c) What jurisdiction-specific requirements apply? This determines whether the platform is legal to operate.
P0
3. Latency Budget Not Defined — Desktop is 100x Faster
Current desktop: signal → price check → execute in <200ms via direct broker WebSocket + API. Proposed SaaS: signal → queue → parse → queue → subscribe resolve → queue → conditional monitor → queue → SnapTrade execute. Each queue hop adds 50-200ms. Total estimated latency: 1-5 seconds. For options trading where prices move 10-50% in seconds, this can mean fills at materially worse prices.
Execution Quality
Remedy: Define latency SLAs per stage: signal parse <50ms, trigger detection <100ms, execution placement <500ms end-to-end. Use in-process event bus (not external queue) for hot path. Reserve external queues for audit/notification/billing cold path. Consider co-locating execution workers with SnapTrade/broker endpoints.
P1
4. Market Data Redistribution Legality
Using "admin broker connections" (Webull, Schwab, Tastytrade) to stream market data and redistribute to platform users likely violates broker terms of service and exchange data licensing agreements. NYSE/NASDAQ real-time data requires a market data vendor agreement ($5-20K/month+) for redistribution. Using a personal broker account to serve platform data is explicitly prohibited.
Blocker Legal Risk
Remedy: For production: license market data from an authorized vendor (Polygon.io, Alpaca data API, IEX Cloud, or direct exchange feeds). For MVP: use delayed/free-tier data for dashboard display, use SnapTrade's own quote endpoints for execution-path pricing, or only trigger based on user's own broker data.
P1
5. Thundering Herd on Signal Broadcast
When a popular signal fires (e.g., 1,000 users subscribed to a channel), the system must simultaneously: create 1,000 conditional orders, monitor all, and when triggered, place 1,000 orders through SnapTrade within seconds. SnapTrade rate limits, broker rate limits, and queue throughput will create a bottleneck. Users 1-50 get filled at the signal price; users 500-1,000 may get filled 5-15% worse (slippage).
Fairness Risk
Remedy: Design fair-execution policy: randomize execution order per batch, implement execution windows, document expected slippage at scale, cap channel subscriber limits. Consider "copy-trading fairness" regulations. At 1,000+ subscribers, this transitions from "automation tool" to "copy trading platform" with different regulatory requirements.
P1
6. 20 Microservices for MVP Is Operationally Untenable
The document proposes 20 independent microservices. For a team of 3-8 engineers, this means: 20 deployment pipelines, 20 health monitors, 20 log streams, 20 sets of inter-service contracts, distributed tracing across 20 services, and cascading failure debugging. Industry consensus: microservices-first is an anti-pattern for teams under 50 engineers.
Architecture Debt
Remedy: Start with a modular monolith (3-4 deployable units) with clean internal module boundaries. Split into microservices when a specific service needs independent scaling. Suggested MVP units: (1) API + Auth + Tenants, (2) Signal Ingestion + Parsing + Conditional Orders, (3) Execution + Position Sync + Risk Engine, (4) Market Data Hub. Extract individual services only when load demands it.
P1
7. SnapTrade Rate Limits at Scale Not Quantified
SnapTrade API rate limits are not addressed in the architecture. If SnapTrade allows 100 requests/minute per API key, serving 10,000 users with 5-second position polling would require 120,000 requests/minute — 1,200x over the limit. The architecture assumes unlimited SnapTrade throughput.
Scalability Ceiling
Remedy: Contact SnapTrade for enterprise rate limits and pricing. Design batch-polling endpoints if available. Implement per-user poll frequency tiers (active positions = 5s, idle = 60s). Use SnapTrade webhooks for push-based position updates instead of polling. Build a SnapTrade connection pool with intelligent request distribution.
P2
8. No Disaster Recovery for Open Positions
If the risk engine goes down while users have open positions with no broker-native stops, those positions are unprotected. The document doesn't define an RTO (Recovery Time Objective) for the risk engine or a failover strategy for position protection during outages.
Financial Risk
Remedy: Define RTO <30 seconds for risk engine. Always place broker-native stop-loss orders as the primary protection layer — software risk engine is the enhancement layer, not the safety net. If the platform goes down, broker-native stops protect capital. Require broker bracket support as a condition for auto-trading.
P2
9. No Testing or Paper Trading Strategy
The architecture has no section on: paper trading simulation, backtesting, integration testing with brokers, or how to safely test execution logic without real money. In a trading platform, untested code paths can directly lose user money.
Safety Risk
Remedy: Add paper trading mode per user (not just per platform). Build a broker simulator service for integration tests. Require automated regression testing for risk engine logic. Signal replay testing: replay historical signals through the system and validate outcomes. All risk engine changes must pass against a test suite of 100+ historical position scenarios.
P2
10. Options Symbol Normalization Complexity Underestimated
Options symbols vary wildly across brokers: OCC format, broker-specific formats, index options (SPX/SPXW/XSP), weekly vs monthly expirations, adjusted strikes. The document mentions "symbol normalization" but doesn't address: expired options cleanup, strike/expiry matching across brokers, or the combinatorial explosion of options chains.
Implementation Complexity
Remedy: Build a dedicated SymbolCanonicalizer service. Use OCC standard as the canonical format. Build broker-specific adapters for translation. Your existing desktop code already handles SPX↔SPXW and NDX↔NDXP — port this logic to a shared library.
P2
11. No Mobile Experience Planned
Trading users expect mobile alerts and basic controls (pause trading, view positions, close positions). The architecture only describes a web dashboard. No mobile app, PWA, or responsive mobile design is mentioned.
Market Fit
Remedy: Design the web dashboard as mobile-responsive from Day 1. Add push notifications (FCM/APNS) for critical events (fills, risk exits, circuit breakers). Consider a React Native or Flutter app in Phase 3. MVP: responsive web + Telegram/Discord notifications covers 80% of mobile needs.
P2
12. Frontend WebSocket/SSE Architecture Missing
Dashboard pages show "real-time updates needed" but the architecture has no WebSocket gateway, SSE endpoints, or real-time data push mechanism for the frontend. Without this, the dashboard will require polling, creating stale data and poor UX.
UX Debt
Remedy: Add a WebSocket gateway service that subscribes to internal event topics and pushes to authenticated user sessions. Use Socket.IO or native WebSocket with JWT auth. Fan-out: tenant_id + user_id scoped channels. Alternatively, start with SSE (Server-Sent Events) for simpler one-directional updates.
P3
13. Signal Provider Performance Verification
Signal providers can publish signals, but there's no mechanism to track and display signal provider performance (win rate, avg return, max drawdown). Without this, users can't evaluate which signal providers to subscribe to.
Product Gap
Remedy: Build a signal_performance table that tracks per-signal outcomes (entry price, exit price, PnL, hold time). Display aggregated metrics per signal provider on the subscription page. Note: displaying performance may have regulatory implications (past performance disclaimers).
P3
14. Multi-Region Not Addressed
The architecture mentions "India" and "Canada" markets but all deployment is single-region. For India/Canada users, latency to US-hosted services may be 200-400ms additional. No data residency or GDPR geographic requirements addressed.
Future Scale
Remedy: For MVP, single US region is fine. Design database schema with region awareness. Plan for multi-region read replicas in Phase 4. India market may require local data hosting for SEBI compliance.

Architecture Strengths

What the document gets right — and what should be preserved.

✓ Domain Expertise

  • + Risk engine evaluation priority is correctly preserved from desktop (Hard SL → Dynamic SL → EMA → Giveback → Trailing → PTs)
  • + Breakout reset guard is a rare but critical feature most platforms miss
  • + Deferred SL re-placement after partial exits handles a real-world edge case
  • + Wide spread guard prevents catastrophic SL fills in illiquid options
  • + Signal replay protection prevents duplicate trades from edited Discord messages

✓ Security & Isolation

  • + Admin vs user broker connection separation is the correct design
  • + Tenant isolation via tenant_id on every query is industry standard
  • + SnapTrade secret encryption via KMS/Vault is the right approach
  • + RBAC with 8 roles covers all stakeholder types
  • + Idempotency design with composite keys prevents duplicate orders

✓ Operational Readiness

  • + Circuit breakers at 7 levels (platform → tenant → user → channel → broker → symbol → market data)
  • + Comprehensive audit logging design covers all sensitive actions
  • + Event-driven architecture enables loose coupling and async processing
  • + Position risk state design faithfully preserves PositionCacheEntry from desktop
  • + Manual trade protection (bot_managed flag) prevents accidental risk management of user's manual positions

✓ Product Completeness

  • + Conditional order state machine (13 states) handles all real-world scenarios
  • + Multi-tier profit targets with custom trim quantities is a differentiator
  • + Hybrid risk monitoring (poll + streaming) is the correct architecture
  • + Notification design covers 19 event types across 6 channels
  • + Phased migration plan (5 phases) is realistic and well-sequenced

Final Recommendation

Executive summary of the architect review.

Do This

  • ✓ Phase 0 first: Validate SnapTrade + legal before writing a line of code
  • ✓ Modular monolith: Start with 3–4 deployable units, not 20 microservices
  • ✓ Software-first risk: Don't depend on SnapTrade for OCO — build software brackets
  • ✓ Licensed market data: Use Polygon.io or Alpaca Data API, not admin broker streams
  • ✓ In-process hot path: Signal → trigger → execute in-process, not through 4 queues
  • ✓ Port the risk engine intact: Your 9K-line risk engine is battle-tested — port, don't rewrite
  • ✓ Paper trading mode: Every user starts in paper mode, graduates to live
  • ✓ Broker-native SL as safety net: Always place broker stop-loss orders; software risk is the enhancement

Don't Do This

  • ✗ Don't start with 20 microservices — you'll spend more time on infra than product
  • ✗ Don't assume SnapTrade can do OCO — validate first, then decide architecture
  • ✗ Don't use admin broker accounts for market data — violates ToS, won't scale
  • ✗ Don't skip the legal review — this could be an RIA/broker-dealer issue
  • ✗ Don't build all 40+ tables on Day 1 — start with ~15 core tables, add as needed
  • ✗ Don't route the hot path through external queues — latency kills options trading
  • ✗ Don't rewrite the risk engine from scratch — the desktop version handles edge cases a rewrite will miss
  • ✗ Don't target 100K users in v1 — design for 1K, architect for 10K, dream about 100K

Bottom Line

This architecture document is an excellent north-star reference — it's one of the most thorough SaaS trading platform designs I've reviewed. The domain knowledge (risk engine, OCO lifecycle, conditional orders, multi-broker execution) is genuinely strong and represents real production learning.

The gap is between design completeness and execution realism. Right-size the MVP to a modular monolith, validate SnapTrade before committing, get legal clearance, and license market data properly. With these adjustments, you can ship a working SaaS to 100 beta users in 6–9 months and scale to 5,000+ users within 18 months.

Estimated total investment to GA: $250–400K over 8–12 months with a team of 4–6 engineers. Break-even: ~500 paying users at $30/month ($15K MRR). Feasible? Yes, with Phase 0 validation and right-sized scope.
