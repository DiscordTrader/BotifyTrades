Module Breakdown

Core Module Architecture

Nine purpose-built modules under src/brokers/webull_official/, each with a single responsibility.

config.py

Config

WebullConfig dataclass holding API credentials, environment routing, and MQTT connection details.

  • ✓ app_key & app_secret
  • ✓ Environment: production / test
  • ✓ Base URL routing (prod vs UAT)
  • ✓ MQTT host & port properties

auth.py

Security

HMAC-SHA1 per-request signing engine for Webull v2 API specification.

  • ✓ UTC timestamp generation
  • ✓ UUID nonce per request
  • ✓ Canonical header sorting
  • ✓ MD5 body hash for POST
  • ✓ Base64-encoded HMAC signature

client.py

HTTP

Async HTTP client wrapping httpx with automatic signing and rate limiting.

  • ✓ httpx.AsyncClient (30s timeout)
  • ✓ 20 max connections, 10 keepalive
  • ✓ Auto sign_request() on every call
  • ✓ Rate limiter acquire before send
  • ✓ Error-class routing (401/order/generic)

rate_limiter.py

Throttle

Per-category sliding-window rate limiter matching Webull's published limits.

  • ✓ Orders: 600 req / 60s
  • ✓ Account data: 2 req / 2s
  • ✓ Token: 10 req / 30s
  • ✓ Subscribe: 600 req / 60s
  • ✓ Async lock + monotonic clock

accounts.py

API

Account discovery and balance queries. Returns typed models.

  • ✓ list_accounts() → WebullAccount[]
  • ✓ get_balance() → WebullBalance
  • ✓ Multi-format response parsing

orders.py

API

Complete order lifecycle: place, cancel, replace, query — for stocks, options, and brackets.

  • ✓ Stock / Option / Bracket / Trailing
  • ✓ Cancel & Replace (modify)
  • ✓ Open orders & history queries
  • ✓ UUID client_order_id generation

positions.py

API

Position fetching with option leg parsing.

  • ✓ get_positions() → WebullPosition[]
  • ✓ Equity & option position support
  • ✓ Strike, expiry, type extraction

models.py

Data

Typed dataclasses for all API response structures.

  • ✓ WebullAccount (id, type, class, label)
  • ✓ WebullBalance (12 financial fields)
  • ✓ WebullPosition (14 fields + option data)
  • ✓ WebullOrder (16 fields + combo type)
  • ✓ PlaceOrderResult (order IDs)

streaming.py

Real-time

MQTT market data streaming and trade event polling for fill detection.

  • ✓ WebullMarketStream (MQTT)
  • ✓ TradeEventPoller (3s fill detection)
  • ✓ Batch subscribe (100 symbols/batch)
  • ✓ Event callbacks (fill, terminal)

Real-Time Data

Market Streaming & Trade Event Polling

Dual-channel real-time data: MQTT for market prices and a 3-second polling loop for trade fill detection.

WebullMarketStream

MQTT
  • ✓ Protocol: MQTT v3.1.1 over TCP:1883
  • ✓ Topics: snapshot, quote, notice, echo
  • ✓ Sub types: SNAPSHOT, QUOTE, TICK
  • ✓ Batch size: 100 symbols per subscribe call
  • ✓ Session ID: UUID hex, unique per connection
  • ✓ Callbacks: on(event, callback) pattern
  • ✓ Auto-reconnect: Disconnect event emitted
// Subscribe request body { "session_id": "a1b2c3d4...", "symbols": ["AAPL", "MSFT", ...], "category": "US_STOCK", "sub_types": ["SNAPSHOT"], "grab": true }

TradeEventPoller

Fills
  • ✓ Interval: 3.0 seconds (configurable)
  • ✓ Fill detection: Compares filled_quantity against tracked state
  • ✓ De-duplication: _known_fills dict prevents re-emit
  • ✓ Terminal events: FILLED, CANCELLED, FAILED
  • ✓ Error resilience: Catches exceptions, continues polling
  • ✓ Graceful stop: CancelledError suppression
// Fill event emitted { "client_order_id": "...", "symbol": "AAPL", "side": "BUY", "filled_qty": 50, "filled_price": 150.25, "status": "FILLED", "new_fills": 50 }
