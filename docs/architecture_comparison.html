⚠️ Where It Breaks Down

Gap 1: No REST-Only Detection

_dirty_symbols is cleared every cycle. There is no persistent tracking of "this symbol has NEVER received a streaming tick." The system assumes if Webull's is_streaming() is true, all symbols are streaming — but MQTT silently ignores illiquid stocks.

Gap 2: Global REST, Not Per-Symbol

REST refresh uses a single boolean _force_webull_rest_refresh every 3-5s for ALL positions. There's no way to poll one symbol at 1s and another at 5s. REST-only symbols like FCHL get the same slow 3-5s interval as streamed symbols.

Gap 3: Stuck ≠ Slow

The stuck price detector triggers when price is literally frozen (same value for 3s). FCHL's price was changing every REST poll — it was slow, not stuck. The detector kept resetting tracker['last_changed'] on every new price.

Gap 4: Schwab-Only Cross Subscribe

_request_cross_broker_subscriptions() only subscribes equities on Schwab hub. It only activates for brokers flagged as "non-streaming" — but Webull is flagged as streaming even when MQTT ignores specific symbols.

✅ What UPH Does Well

Single Cache, All Brokers

One get_quote() call checks all broker hubs automatically. No need to query Webull, then Schwab, then IBKR manually.

Built-in Freshness

Every quote has freshness tag: fresh (≤3s), aging (≤5s), stale (≤10s), degraded (≤30s), unverified. Conditional orders could use this instead of accepting any price.

Eliminates ~300 Lines

Risk engine's _update_prices_from_hub(), _try_cross_hub_price(), and conditional order's _try_streaming_hub_price() all duplicate what UPH does centrally.

❌ Why UPH Alone Doesn't Fix FCHL

UPH Can Only Aggregate What Exists

FCHL's problem wasn't which layer aggregates the data — it was that no broker hub was streaming FCHL at all. Webull MQTT silently ignored this penny stock. Schwab WebSocket wasn't subscribed to it. UPH subscribes to hub events, but if no hub emits an event for FCHL, UPH's cache is equally empty.

UPH would still fall back to REST polling at 3-5s intervals — the same blind spot that caused the gap-through.

🚨 Three Critical Concerns Addressed

Concern: 10s Detection Too Slow

10s means you could miss PT entirely on a quick spike, or SL on a fast drop. A trade that lasts 15s has NO fast-poll protection for the first 10s.

Solution: 3-Second Detection
Monitoring cycle runs every ~1s. After 3 cycles with zero streaming ticks while Webull is_streaming() is globally true, the symbol is classified REST-only. This matches the existing stuck price threshold (3s) — proven safe in production.

Concern: API Rate Limit Exhaustion

Blind 1s polling burns rate limits fast:

Webull30 req / 60s= 0.5/s (1s poll = 2x over!)
Schwab120 req / 60s= 2/s (safe for 1-2 symbols)
IBKR50 req / 1s= 50/s (no concern)
Solution: Rate-Limit-Aware Smart Polling
Check RateLimitManager before every REST call. Distribute across brokers (Schwab first → Webull fallback). Dynamic interval: 2s per symbol (Schwab budget allows ~2 REST-only symbols safely). Skip if budget exhausted — never 429.

Concern: How Does Streaming Auto-Recover?

When Schwab subscribes to FCHL, how does the system detect it's now streaming and stop wasting REST calls?

Solution: Instant Self-Healing via Existing Callback
ALL hubs share the same _on_quote_update() callback. The moment Schwab WebSocket pushes a FCHL tick → callback fires → updates last_tick → clears rest_only_since → next cycle: symbol is no longer REST-only → fast polling stops → sub-50ms streaming path takes over.

🔧 The Five Changes

Change 1: Per-Symbol Streaming Status (3s Detection)

position_monitor.py

New persistent dict tracks streaming status per symbol. Detection fires at 3 seconds — same as the proven stuck price threshold. Self-heals instantly when any hub delivers a streaming tick.

# New data structure in __init__ (persistent, NOT cleared per cycle) self._symbol_streaming_status = {} # {symbol: {first_seen, last_tick, rest_only_since}} self._REST_ONLY_DETECTION_DELAY = 3 # 3s — matches stuck price threshold # In _on_quote_update callback (hub thread) — SELF-HEALING: with self._symbol_streaming_status_lock: status = self._symbol_streaming_status.get(symbol) if status: status['last_tick'] = tick_ts # streaming tick arrived! status['rest_only_since'] = None # clear REST-only → stops fast polling
3s detection (was 10s) Thread-safe Stocks + options Instant self-healing

Change 2: Rate-Limit-Aware Smart Polling

position_monitor.py

Instead of blind 1s polling that burns rate limits, checks RateLimitManager before each call. Uses Schwab REST first (2 req/s budget), falls to Webull only if Schwab exhausted. Dynamic interval: 2s per symbol to stay within Schwab's 120/min limit safely.

# Rate-limit-aware polling (NOT blind 1s for every symbol) self._REST_ONLY_POLL_INTERVAL = 2.0 # 2s per symbol — Schwab budget: 120/min = 2/s self._REST_ONLY_MAX_PER_CYCLE = 2 # max 2 symbols per 1s cycle async def _poll_rest_only_symbols(self, positions): for pos in targets[:self._REST_ONLY_MAX_PER_CYCLE]: if (now - last_poll) < self._REST_ONLY_POLL_INTERVAL: continue # respect per-symbol cooldown if not rate_manager.can_make_request('schwab'): continue # skip if rate budget exhausted — never 429! fresh = await self._try_rest_quote(pos) pos.current_price = fresh
Rate Budget Math:
Schwab budget:120 req/min = 2 req/s
Normal usage (positions, orders):~20-40 req/min
Available for REST-only polling:~80 req/min = 1.3 req/s
2 REST-only symbols at 2s each:60 req/min ✓ within budget
5 REST-only symbols at 2s each:150 req/min — rate manager throttles automatically
Rate-limit checked before each call Schwab-first (higher budget) Never triggers 429 2s interval (was 1s)

Change 3: Faster Schwab Subscription Drain (10s → 3s)

schwab_streaming_client.py

Currently Schwab streaming client checks for pending subscriptions every 10 seconds. That means after we request FCHL subscription, it takes up to 10s before the WebSocket actually subscribes. Reducing to 3s means streaming starts much sooner — and once it does, self-healing kicks in immediately.

# schwab_streaming_client.py line 409 # Before: if now - self._last_pending_drain >= 10: # checks every 10s — too slow! # After: if now - self._last_pending_drain >= 3: # checks every 3s — streaming starts sooner
10s → 3s drain interval Streaming starts 7s sooner One-line change

Change 4: Proactive Schwab Subscription on Detection

position_monitor.py

The moment a symbol is classified REST-only, immediately request Schwab WebSocket subscription. Combined with the 3s drain (Change 3), streaming can start within ~6 seconds of position open (3s detection + 3s drain). Once Schwab streams the first tick, self-healing (Change 1) switches to sub-50ms path.

# Inside _poll_rest_only_symbols — right after classification equity_rest_only = {sym for pos in positions if pos.symbol.upper() in rest_only_syms and pos.asset != 'option'} new = equity_rest_only - self._rest_only_schwab_subscribed if new: self._request_cross_broker_subscriptions(new) # existing method! self._rest_only_schwab_subscribed |= new
Equities: Schwab WebSocket subscribe Options: REST-only fast polling (no subscribe API) Reuses existing method

Change 5: Conditional Order Freshness Gate

conditional_order_service.py

Currently accepts any cached price regardless of age. A 60-second-old price can trigger a conditional order. New gate: reject hub prices older than 5 seconds, fall through to REST.

# Before (accepts ANY cached price — could be 60s old!): price = hub.get_quote_price(symbol) if price and price > 0: return price # After (freshness-gated — max 5s old): quote = hub.get_quote(symbol) if quote and (time.time() - quote.timestamp) <= 5: return quote.last # fresh price # else: falls through to REST (existing behavior)
5s freshness gate Falls through to REST if stale Zero breaking changes

🎯 Decision Framework

UPH Alone = Wrong Tool for This Problem

UPH is an aggregation layer — it centralizes cross-hub queries and adds freshness metadata. It's excellent for code quality but cannot create data that doesn't exist. For FCHL, no hub was emitting events → UPH cache is equally empty → same 3-5s REST gap.

Adaptive REST = Targeted Fix

Addresses the root cause: detecting that a symbol lacks streaming and acting on it. Per-symbol tracking → faster polling → proactive subscription → self-healing. All additive — zero changes to existing code paths. Works for both stocks and options.

🔮 Future: UPH + Adaptive REST Together

The ideal long-term architecture combines both: Adaptive REST for the detection and fast-polling engine, and UPH as the centralized price source that replaces the 300+ lines of scattered cross-hub logic. UPH's freshness classification feeds into the REST-only detector. Adaptive REST's targeted polls feed back into UPH's cache.
