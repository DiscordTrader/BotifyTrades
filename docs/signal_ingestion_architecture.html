⚠️ The Problem

Why the Current Approach Must Change

Discord selfbots worked in 2023. In 2026, they're a ticking time bomb for a commercial product.

🚫 Current: Discord Selfbot Risks

  • Violates Discord Terms of Service — account ban risk
  • User token can be invalidated anytime (password change, Discord security sweep)
  • Token extraction via DevTools is confusing for non-technical users
  • Commercial distribution of selfbot tools increases legal exposure
  • Discord actively detects and terminates automated user accounts
  • No official support or documentation — library can break on any Gateway update

✅ New: Multi-Source Signal Platform

  • 100% Discord ToS compliant — no user token needed
  • Multiple signal sources: webhooks, bot relay, Telegram, direct API
  • Professional onboarding — no DevTools knowledge required
  • Signal providers can officially integrate via partnership
  • Cloud relay ensures reliability — no local network requirements
  • Fully documented, supported, and commercially viable

🔐 Why Username + Password Login Is Not Possible

Discord removed programmatic username/password authentication. There is no API endpoint that accepts credentials and returns a usable session. Here's what happened:

2020 POST /api/auth/login accepted email + password, returned user token. Automated login was possible. 2021 Discord added mandatory hCaptcha to the login endpoint. Automated requests receive captcha_key challenges. 2022 MFA/TOTP enforcement expanded. Even with CAPTCHA bypass, 2FA blocks automated auth flows. 2023 Discord's anti-automation detection flags programmatic login patterns. Accounts get locked for "suspicious activity." 2024+ OAuth2 is the only sanctioned auth path — but no OAuth2 scope grants message reading ability. The experimental messages.read scope was restricted and removed from general availability.
💡
Bottom line: Even if you could bypass CAPTCHA + MFA programmatically (which would be fragile, unreliable, and still violate ToS), Discord would detect and ban the account. The correct solution is to not authenticate as the user at all — instead, use an official Discord Bot or webhook relay.
🏗️ Architecture

The Desktop-First Signal Platform

Each user runs their own BotifyTrades instance. The cloud relay is a thin signal bridge — all execution stays local.

🖥️
Key Constraint: BotifyTrades is a desktop application. Every user runs their own instance with their own broker credentials. Unlike cloud SaaS, we cannot run one bot that serves all users. The architecture must account for individual desktop apps connecting to a shared signal relay.

Signal Ingestion Architecture v2.0 — Complete Data Flow

Complete signal flow: Sources → Cloud Relay → Desktop App → Broker Execution
💡
The Key Insight: The cloud relay is not a trading platform. It never sees broker credentials, never places orders, never holds positions. It's a thin message router — it receives signals from various sources and forwards them to subscribed desktop apps via WebSocket. All execution, risk management, and broker interaction stays on the user's local machine. This keeps the architecture simple, secure, and regulation-friendly.

Tier 1 — Build Now

Cloud Webhook Gateway

A lightweight cloud endpoint that receives signals from TradingView, Zapier, or any HTTP source, then relays to subscribed desktop apps.

  • 🔗 POST /api/v1/signals — Standard REST endpoint accepts JSON signals from any source
  • 🔑 Each user gets a unique API key tied to their license — rate-limited, authenticated
  • 📡 Cloud relay pushes signal to user's desktop app via persistent WebSocket
  • 🔄 TradingView-compatible format — users paste their webhook URL in TV alert settings
  • 🧹 Server-side deduplication — same signal from multiple sources only executes once
  • ⚡ No NAT/firewall issues — WebSocket is outbound from user's machine
How It Works

User Setup Flow

Three clicks to connect. No Discord token, no DevTools, no port forwarding.

  • 1️⃣ User opens BotifyTrades desktop app → Settings → Signal Sources
  • 2️⃣ App auto-generates a personal webhook URL: https://relay.botifytrades.com/hook/usr_abc123
  • 3️⃣ User pastes this URL in TradingView alert settings, Zapier, or gives it to their signal provider
  • 4️⃣ Signals flow: Source → Cloud Relay → WebSocket → Desktop App → Broker

Webhook API — Signal Format

// POST https://relay.botifytrades.com/api/v1/signals // Header: Authorization: Bearer usr_abc123_key { "action": "BTO", // BTO, STC, BTO_OPTION, STC_OPTION "symbol": "SPY", "strike": 580, // Options only "expiry": "2026-05-02", // Options only "opt_type": "C", // C or P — options only "price": 2.50, // Limit price (null = market) "source": "tradingview", // Source identifier "channel": "my-alerts" // Maps to channel risk settings } // TradingView-compatible format (auto-detected): "BTO SPY 580C 05/02 @2.50" // Raw text — parsed by signal pipeline // Response: { "status": "relayed", "signal_id": "sig_xyz789" }
📋
The webhook gateway accepts both structured JSON and raw text. Raw text goes through the same signal parsing pipeline used for Discord messages — all 7 parser tiers. This means any format that works in Discord also works via webhook.

🔀 Provider Options

Three Ways Providers Can Bridge Signals

All three feed into the same cloud relay. The user's desktop app never knows which one the provider chose.

🅰️ Cloud-Hosted Bridge

BotifyTrades runs the selfbot on the cloud relay server. Provider just adds a dedicated Discord account to their server with read access. Easiest for providers.

Provider effort: Add 1 account (2 clicks)
BotifyTrades effort: Run + monitor selfbot
If banned: New account, provider re-adds (~5 min)
ToS risk: 1 disposable account

🅱️ Provider-Hosted Bridge

Provider runs a lightweight Python script on their own server. The script reads signals and POSTs to the BotifyTrades webhook. Provider owns the risk entirely.

Provider effort: Run a Docker container
BotifyTrades effort: Provide turnkey container
If banned: Provider handles it — BotifyTrades is clean
ToS risk: Provider's choice

🅲 Provider's Own Micro-Bot

Provider runs a 20-line Python script as their own bot — they control the token, the code, and can shut it down anytime. 100% ToS compliant.

Provider effort: Create bot at discord.com/developers, run 20-line script
BotifyTrades effort: Provide the script template
If issues: No ban risk — official Discord bot API
ToS risk: Zero for everyone
🔑
The key insight: All three provider approaches POST signals into the same relay ingest endpoint. The cloud relay doesn't care how the signal arrived — selfbot bridge, provider script, or micro-bot. And the user's desktop app definitely doesn't care — it just receives parsed signals via WebSocket. This means providers can start with Option A (easiest) and migrate to Option C (safest) anytime, with zero changes on the user side.

Discord Webhooks are incoming only. They let you POST messages into a Discord channel — not out of it. There is no "outgoing webhook" in Discord (unlike Slack). Every tool that reads messages OUT of Discord does so by adding a bot to the server.

Recovery Plan

What If the Bridge Account Gets Banned?

The bridge account is disposable by design. It has no personal data, no chat history, no servers it owns. Recovery is a 5-minute process.

  • 1️⃣ Detection: Cloud relay detects disconnect — alerts BotifyTrades ops via webhook/email
  • 2️⃣ Create new account: New Discord account created (email + password — just for this purpose)
  • 3️⃣ Provider re-adds: Provider admin adds the new account to their server (2 clicks)
  • 4️⃣ Auto-resume: Cloud relay restarts selfbot with new token — signals resume automatically
  • ⏱️ Total downtime: 5-30 minutes depending on provider response time
  • 🔒 User impact: ZERO. No user account is affected. Existing positions and risk management continue locally
Risk Mitigation

How to Minimize Ban Risk

Read-only selfbots have the lowest detection risk. These practices make the bridge account virtually invisible to Discord's anti-abuse systems.

  • 👁️ Read-only: Never send messages, reactions, or typing indicators — pure passive reading
  • 🐌 Low activity: Only connect to signal channels, not general chat — minimal Gateway events
  • 🔄 Natural patterns: Mimic real user behavior — connection times, heartbeat intervals
  • 🏠 Dedicated IP: Cloud relay has a static IP — no VPN hopping that triggers flags
  • 📱 Account aging: Let accounts age 1-2 weeks before using as bridge — fresh accounts draw scrutiny
  • 🎭 Multiple backup accounts: Have 2-3 accounts ready — instant swap on ban, zero downtime

🔒 Security

Security Architecture

Broker credentials stay local. The cloud relay is a dumb pipe.

🔐

Credentials Never Leave Desktop

Broker API keys, tokens, and passwords are stored locally in encrypted SQLite. The cloud relay has zero access to broker credentials.

🎫

License-Based Auth

WebSocket connection authenticated via license key + machine fingerprint. No Discord credentials involved.

🔄

End-to-End TLS

WebSocket (wss://) and HTTPS — all traffic encrypted in transit. Signal content is visible only to the cloud relay (for routing) and the user's desktop.

🚫

No Order Execution in Cloud

The cloud relay cannot place orders, modify positions, or interact with brokers. It only forwards signal text.

📋

Audit Trail

Every signal relayed is logged with timestamp, source, and delivery confirmation. Users can review signal history in the desktop app.

⚡

Stateless Relay

Cloud relay stores no trade data, no positions, no P&L. If the relay goes down, desktop apps continue trading with existing positions and risk rules.

❓ FAQ

Common Questions

"What if the provider doesn't add the bot?"

Users can still use Tier 1 (Webhook API). Many signal providers already support TradingView alerts, Zapier, or direct webhook output. Users can also set up their own bridge using tools like n8n or IFTTT to relay messages from any source to the BotifyTrades webhook.

"What about the MESSAGE_CONTENT intent?"

Discord requires the MESSAGE_CONTENT privileged intent for bots in 100+ servers. This requires verification (bot description, privacy policy, intended use). BotifyTrades qualifies as a legitimate use case — trade signal forwarding with user consent. Apply early in Phase 2.

"What if the cloud relay goes down?"

Existing positions and risk management continue running locally. The desktop app has full autonomy — SL/PT/OCO brackets are already placed at the broker. Only new signal ingestion pauses. The WebSocket client auto-reconnects with exponential backoff.

"Does the user need to be in the Discord server?"

No. That's the breakthrough. The bot is in the server — the user doesn't need to be. Users who don't even have Discord can still receive signals from Discord providers via the relay. This dramatically expands the addressable market.

"Can I still use the selfbot?"

The selfbot mode will be maintained as a legacy option for users who understand the risks. It feeds into the same signal parsing pipeline. Over time, as providers onboard the official bot, selfbot usage will naturally decrease.

"What's the cost of the cloud relay?"

A single $20/month VPS handles thousands of WebSocket connections and the Discord bot. Redis for pub/sub adds ~$5/month. Total infrastructure cost: ~$25/month — trivially covered by even a few user licenses.
