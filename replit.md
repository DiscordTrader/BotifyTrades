# BotifyTrades — Compressed Documentation

## Overview

BotifyTrades is a production-grade, cross-platform automated trading bot designed to monitor Discord and Telegram for trade signals and execute them across multiple brokers for US and Canadian markets. It features a comprehensive Flask-based web control panel with real-time dashboards, advanced risk management, WebSocket streaming, AI-powered analysis, and detailed portfolio analytics. The project aims to provide automated trading capabilities, offering users a powerful tool for managing and optimizing their trading strategies.

## User Preferences

- **Security**: Always use environment variables (Replit Secrets) for credentials and license keys
- **Testing**: Test with paper_trade = true before enabling live trading
- **Monitoring**: Review console logs regularly for trade execution
- **Channel filtering**: Only process signals from designated channels
- **Deployment**: Prefer local machine or cloud VPS for 24/7 operation
- **Licensing**: All deployments require a valid license key (set via LICENSE_KEY environment variable or setup wizard)
- **Authentication**: First-time users are guided through setup wizard to create admin account with email recovery

## System Architecture

### UI/UX Decisions
The web control panel, built with Flask, provides real-time dashboards for broker status, live positions, P&L, and risk statuses. It includes detailed trade monitoring, performance analytics, intuitive settings, and visual feedback mechanisms like streaming indicators. A redesigned performance dashboard offers a calendar view with key metrics, daily P&L, trade counts, and cumulative P&L charts.

### Technical Implementations
- **Signal Sources**: Integrates with Discord and Telegram, using a 5-tier parsing system (dedicated parsers, regex, AI fallback) for robust signal detection, deduplication, and follow-up updates.
- **Risk Management Engine**: A multi-layered system with global and per-channel settings, configurable Stop Loss, Trailing Stop, four Profit Targets, Dynamic SL Escalation, Max Profit Giveback Guard, EMA-5 Candlestick Risk Engine, and a "Leave Runner" feature. Risk states are persistent and exit pricing prioritizes streaming hub bid/ask. `get_channel_risk_settings()` includes a direct channel lookup fallback when the LEFT JOIN (trades→channels) returns NULL for all channel columns — this prevents channel-managed trades from being incorrectly treated as "[Manual Trade]" with global risk applied. `find_open_bto_trade()` uses case-insensitive broker matching (`LOWER(broker) = LOWER(?)`) to handle WEBULL/Webull casing inconsistencies.
- **Position Sizing**: Per-channel percentage-based sizing uses live broker data, considering specific buying power metrics across brokers and three balance modes: "Live", "Pre-Market", and "Start-of-Day."
- **Order Execution System**: An asynchronous, queue-based system for multi-broker order execution, featuring per-channel broker selection and a Universal Order Placement Resilience Layer with error classification, circuit breakers, and retries. Includes hub-first slippage checks and an Order Chaser service.
- **Order Management System (OMS)**: Handles dynamic SL/PT management, exit order arbitration, position matching, and FIFO-based P&L tracking, supporting 12 types of exit source classifications. It features universal risk exit P&L recording, multi-lot overflow closure, broker fill price reconciliation, and consolidated fill-based P&L.
- **WebSocket Streaming**: Utilizes Webull MQTT and an optimized Schwab WebSocket system for real-time quotes and orders, providing sub-100ms pricing for conditional orders and quick trade options chains. Webull REST quote extraction uses `_extract_current_price()` helper with priority: bid/ask midpoint → pPrice (pre-market) → last → close (final fallback only), ensuring pre-market/extended hours prices are used instead of yesterday's close. Options chain UI uses poll-first architecture (500ms interval to `/api/options/stream-quotes`) with optional SSE upgrade via `/api/options/chain-stream`. Includes cross-broker streaming subscription (Webull→Schwab OCC key cross-sub) and immediate BTO symbol subscription. Key format normalization: Webull hub stores underscore keys (`SPY_654_C`), Schwab hub stores OCC keys (`SPY   260401C00654000`), frontend uses underscore. All endpoints convert between formats.
- **Daily P&L Limit System**: Per-broker daily P&L tracking against SOD equity snapshots, with configurable dollar and percentage limits for profit targets and loss limits. Max Daily Trades enforcement works independently of the P&L dollar/percentage master toggle — trade counting and locking activate whenever a trade limit is configured, even if the full P&L system is disabled. Stale P&L locks (loss/profit) are normalized when P&L is disabled so they don't block trade-limit enforcement.
- **Broker Sync Service**: Reconciles database state with actual broker states, detecting order fills/cancellations, position changes, and account updates. Adaptive fill sync cadence: every cycle when PENDING/OPEN trades exist, every 5 cycles when idle.
- **Fill Accuracy Architecture**: A five-layer fill-to-PNL pipeline ensures accurate tracking of entry and exit fills. Unified `process_filled_order_event()` in `gui_app/database.py` propagates fills to both `trades` and `signal_lots`/`lot_closures` tables via single idempotent call with `BEGIN IMMEDIATE` transaction safety. Channel-strict matching: all fallback lot matching requires known `channel_id` to prevent cross-channel P&L contamination. Position-disappearance closures using `last_sync` prices are tagged `provisional_sync` and overwritten when deterministic fills arrive.
- **Order Chaser Service**: Monitors unfilled exit/entry orders and replaces stale ones with better prices, tracking all STC orders (risk management and signal STCs).
- **Security & Authentication**: Implements admin account management, password hashing, email recovery, session-based authentication, rate limiting, and encrypted broker credentials, with Schwab OAuth using CSRF state tokens and PKCE. Local password recovery writes code to a server-side file only (never exposed in HTTP responses) — user must have machine access to read the recovery file. Non-existent user requests generate identical UX to prevent username enumeration.
- **Database Architecture**: Uses SQLite with WAL mode for concurrent read/write operations, storing trading data and encrypted credentials. Includes retry mechanisms for critical exit bookkeeping. NOTE: FK enforcement (`PRAGMA foreign_keys=ON`) is intentionally OFF — schema FK definitions reference `channels(id)` (integer autoincrement) but `trades.channel_id` stores Discord channel ID strings. Enabling FKs would break all trade INSERT operations. Schema FK migration required before enforcement can be enabled.
- **AI Analysis**: Integrates OpenAI GPT for pre-trade/post-trade analysis, an AI chat assistant, and AI command toggles.
- **Risk Engine Direct Exit Architecture**: Dual-path exit execution system (primary queue-based, backup daemon thread at 5s for direct calls) with an Exit Lease Manager to prevent duplicate exits.
- **Risk Engine Speed Optimizations**: Lowered interval floor, early-wake sleep chunking, cached service checks, throttled cache saves, parallel broker fetches, REST position cache, hub-first approaches, cross-broker price updates, and a staleness guard.
- **Event-Driven Fill Watch System**: Activates rapid position polling after BTO orders, with broker-specific intervals, detecting new positions or quantity increases.
- **Risk Engine Staleness Gate**: Pre-execution staleness protection for risk exits with REST/cross-hub validated override. States: `_rest_confirmed_this_cycle` (REST returned different price, per-cycle), `_rest_validated_same` (REST or cross-hub confirmed same price is real, 60s TTL). `_try_rest_quote` returns same-price confirmations (with `_last_rest_source='confirmed-same'`) instead of None, enabling `_rest_validated_same` to be set for low-liquidity/unchanged-price stocks. REST-same-price validations don't consume `_MAX_REST_REPAIRS_PER_CYCLE` quota (only actual price-change repairs count). Already-validated positions get +4 extra REST quota for re-validation. Provisional validation at 3s (`_STUCK_PRICE_THRESHOLD`) eliminates blocking gap — validation is set before the 5s blocking threshold activates. 60s absolute escape hatch: if price unchanged >60s and at least one successful REST response during this freeze epoch (`rest_checked_ok > last_changed`), exits force-allowed regardless of validation state. Validation requires passing sanity checks (no 30%+ deviation rejection). Repair cycles allow exit evaluation only when REST-confirmed. Partial exits protected by `_partial_exit_in_flight` (per-tier, 30s TTL) preventing double-trim. Cross-hub zombie detection via `_is_hub_live()` requires actual tick receipt (`_last_quote_ts > 0`) and recency (< 120s). Worker lease is fail-closed (exceptions skip exit instead of proceeding). `dynamic_sl` classified as SL-type for market order routing.
- **Trading212 Data Hub & Integration**: Provides a portfolio-based price cache for T212 positions, integrating with the risk engine and conditional order service, with real-time architecture and cross-broker streaming. LIVE environment capability: Full order type support — Stop, Limit, Stop-Limit, and Market orders all supported on both LIVE and DEMO accounts per official T212 API docs. Risk engine places native broker-side SL/PT brackets for T212 LIVE (stop orders for SL, limit orders for PT). DEMO accounts still use market-only exits for simplicity. Stop-resync places native stop orders on T212 for both LIVE and DEMO. Rate limiter splits order placement POST (1 req/2s) from order polling GET/DELETE (1 req/5s), with `/history` classified before `/orders` to correctly bucket `/equity/history/orders`. Client API paths validated against T212 v0 spec: `GET /equity/positions/{ticker}` (path param), `POST /equity/orders/market` (negative qty = sell), `DELETE /equity/orders/{id}`. Sell convention: `action='STC'` or `action='SELL'` → `qty = -abs(quantity)`. Exit flow: risk engine `_build_stc_signal()` → hub bid/ask pricing → order type per channel settings → worker queue → `_direct_execute_exit` backup (5s) → broker `place_stock_order()` → appropriate order type.
- **IBKRDataHub Streaming**: Provides real-time IBKR position and price streaming via `ib_insync` native events, managing `reqMktData` subscriptions and integrating with the risk engine.
- **Cross-Hub Price Sourcing**: Cascades through multiple data hubs (Webull → Schwab → IBKR → Tastytrade → T212) to find the freshest price for both stocks and options, normalizing symbols and expiry formats. All cross-hub registries (conditional order monitors, unified price hub, risk engine) must include all 5 hubs: Webull, Schwab, IBKR, TastyTrade (DXLink streaming), and Trading212.
- **Unified Price Hub (Phase B - Shadow Mode)**: A singleton service wrapping all existing data hubs into a single aggregated price cache, currently in shadow mode for comparison against the risk engine's prices.
- **Conditional Order Frozen Price Detection**: `StreamingPriceMonitor` detects frozen prices, probing cross-broker streaming hubs and then broker REST APIs, with a robust fallback chain.
- **Execution Idempotency**: Prevents duplicate trade execution using per-order asyncio.Lock, set guards, and database CAS status checks.
- **Monitor Failure Recovery**: Marks orders as ERROR on monitor crash and attempts automatic restart, setting ERROR status with details if restarts fail.
- **Rate Limit Enforcement**: REST fallback calls and frozen probes check and record API calls against a rate limit tracker.
- **Risk Management Parity (Signal Routing Engine)**: Ensures full risk parity between the Position Monitor and Signal Routing Engine, incorporating EMA and early trailing stops into virtual positions.
- **Trading212 Event Loop Mismatch Fix**: Addresses `asyncio.Lock` and `aiohttp.ClientSession` binding issues to ensure proper function of conditional order monitors and position fetching.
- **T212 Conditional Order Race Condition Fix**: Three-layer broker registration (early in on_ready, background in _init_brokers_background, deferred 15s later) ensures T212 and all brokers are registered with the conditional router regardless of startup timing. Removed destructive `clear()` calls in `_retry_start_monitor` that wiped valid broker/hub registrations. Added visible `print()` logging for monitor startup, auto-discovery, and trigger events (previously only logged to stderr).
- **Quick-Swing-Alerts Signal Parser**: Added `QUICK_SWING` format to the signal format registry (`signal_format_registry.py`) for the quick-swing-alerts Discord channel. Handles embed-based signals with `Ticker: $SYM / Entry: range (...) / Target: (...) / Stop loss: Below X` format. Three sub-parsers: entry (creates conditional orders with entry range, SL, targets, share count from comments), cancel, and exit. Entry range parsing strips share counts and sizing markers before extracting prices, with median-based outlier rejection. Routes through the existing protrader conditional order pipeline (`_conditional_order=True`, `_protrader_cancel`/`_protrader_exit` flags). Also added exclusion in `parser.py` `is_conditional_order_signal()` for the `Entry: range (` format to prevent double-parsing.

- **SignalConversationStateManager Method Fix**: All 5 calls to nonexistent `state_mgr.register_signal()` (in phoenix, hengy, equity-genie conditional order flows) fixed to use the correct method `state_mgr.register_signal_context()` with the correct parameter signature.

(1) `find_origin_channel()` Pass 1 (order_id match) and Pass 2 (position-key match) both enforce OPEN/PENDING/PARTIAL status filter + broker match + `_channel_has_broker_enabled` validation — closed/historical trades excluded. (2) Defense-in-depth in `position_monitor.py`: after `get_channel_risk_settings()` resolves, checks `trades.source` — if `source` is `sync` or `risk_auto_import` (auto-imported by broker sync or risk engine), channel settings are BLOCKED. ### System Design Choices
- **Broker Isolation**: Prevents data cross-contamination by isolating each broker's streaming data.
- **Thread Safety**: All shared states are protected by locks.
- **Priority Queue Execution**: Risk management exits are prioritized over normal entries.
- **Unprotected Trades Banner**: Critical dashboard banner to identify trades lacking risk protection.
- **Hub-First Architecture**: Prioritizes cached streaming data over REST API calls.
- **Graceful Degradation**: Automatically falls back to REST polling if streaming services are unavailable.
- **Modular Broker Abstraction**: Uses a common interface for diverse broker APIs (Webull, Alpaca, Schwab, IBKR, Tastytrade, Robinhood, Trading 212).
- **Help Center**: Comprehensive `/help` route with onboarding, broker connection guides, and risk management diagrams.
- **Market Isolation**: Conditional order services for different markets operate independently.
- **KC Trades Parser**: Dedicated embed parser (`src/signals/kc_trades_parser.py`) for KC Trades Discord bot (author_id: `954609923189248053`). Handles options entries (`TICKER STRIKEc/p EXPIRY @ PRICE`), exits (trimmed/out half/out majority/all out/closed/closing/stopped on), adds (added/added to/back in), bare entries (ticker from title, strike in description), and multi-line entries. Supports `@`/`at` price anchors, `+` prefix stripping, DTE/month-name/M/D expiry formats. Skips corrections (`*`), fills, rolls, and commentary. Wired into `selfbot_webull.py` early detection + routing + PNL tracking + forwarding paths.
- **Equity Genie Parser**: Dedicated embed parser (`src/signals/equity_genie_parser.py`) for the Equity Genie Discord channel. Handles OPEN/ENTRY embeds (breakout entries with multi-ticker support, PT extraction across 6+ format variants, price-only entries) and EXIT/CLOSE embeds (actionable exits with partial percentages, celebratory/informational skips). Each ticker = separate conditional order via Hengy model. Sir Goldman conflict guard prevents false matches on overlapping ENTRY/EXIT titles. Integrated into `selfbot_webull.py` early detection scope + routing block.
- **Conditional Order Streaming Architecture**: Uses `StreamingPriceMonitor` and `BrokerPriceMonitor` with a robust fallback chain for price data, including brokerless monitor upgrades.
- **Risk Engine Price Flow**: Features both full cycle and incremental event-driven evaluation paths.
- **Progressive Broker Bracket Orders — Single Owner Architecture (All Brokers)**: Risk Engine is the sole owner of ALL bracket orders across ALL exit modes (risk, hybrid, signal). Entry path (`selfbot_webull.py`) NEVER places bracket orders — only simple entry orders. After fill detection, Risk Engine places initial SL stop + PT1 limit at broker via `_place_initial_broker_bracket()`. Bracket gate removed from all 3 entry paths (main, paper-legacy, live-legacy) and bracket auto-generation removed. Risk Engine gate no longer checks exit_mode — fires for any mode when SL>0 or PT1>0. `get_channel_risk_settings()` now returns settings for signal mode when channel/signal SL/PT exists (global risk OFF fallback). When each PT tier hits, next PT placed at broker via `_place_next_pt_bracket()` and dynamic SL updated via `_sync_stop_to_broker()` cancel+replace. Cascade is idempotent (guarded by `broker_pt_tier` inside per-position lock), and outstanding bracket orders cancelled before full exits. All bracket ops serialized through per-position `_broker_stop_locks`. Failed PT placements retry up to 2x with 1s delay. PT orders registered with order chaser; chaser uses progressive price escalation. `trim_order_mode='market'` suppresses broker-side PT placement. Early trailing syncs stop changes to broker. After each PT cascade, broker stop resized to remaining qty. Broker-specific: IBKR uses ib_insync StopOrder/LimitOrder. TastyTrade uses OrderType.STOP for stocks, no option stops. Trading212 stocks only. Robinhood `order_sell_stop_loss()` for stocks. Webull `place_order(STP/LMT)` stocks only — options monitored locally. See `docs/BRACKET_ORDER_ARCHITECTURE.md` for full documentation.
- **Broker Bracket Mode Toggle**: Per-channel `broker_bracket_mode` setting (`both`|`sl_only`|`pt_only`|`none`) controls which bracket orders are placed on the broker. Default `both` for backward compat. Guards in `_place_initial_broker_bracket()` (skips SL/PT based on mode), `_place_next_pt_bracket_inner()` (skips PT if mode excludes it), `_sync_stop_to_broker_inner()` (skips SL sync if mode excludes it). `ChannelRiskSettings` has `allows_broker_sl` and `allows_broker_pt` properties. UI radio group in channel risk settings with `?` help modal. Useful for Webull options (SL Only mode avoids ORDER_NOT_SUPPORT_REVERSE_OPTION rejection).
- **Partial Exit Bracket Cancel Fix**: Removed `not decision.is_partial` guard from bracket order cancellation before exits — previously, partial exits (PT trims) would not cancel existing broker bracket orders, causing Webull ORDER_NOT_SUPPORT_REVERSE_OPTION errors when combined SL+PT qty exceeded position size. Now cancels broker brackets before ALL exits (full and partial) with selective cancellation (`cancel_stop`/`cancel_pt` params on `_cancel_broker_bracket_orders`), allowing proper order resizing after trim. Signal routing risk settings inherit `broker_bracket_mode` from source channel. Stale cache cleanup detects orphaned bracket orders and logs them. Settings cache invalidation includes `broker_bracket_mode` in `risk_fields` set.
- **Exit Dedup Hardening (6-fix)**: (1) Worker no longer calls `release_exit_marker` on STC order placement — marker held until fill confirmed, preventing backup thread from firing a duplicate STC within the 5s window. (2) GUI manual close (`close_position_by_id`) now acquires exit lease + sets closing flag + cancels outstanding bracket orders before placing broker close orders. (3) 180s closing timeout in `is_closing()` now checks for active chaser orders before resetting — if chaser still tracking an order for the position, closing flag stays set. (4) `clear_closing` calls fixed to `reset_closing` (correct method name). (5) `_place_initial_broker_bracket` now serialized under `_broker_stop_locks`. (6) Alpaca `place_stock_order` fixed to treat price<=0 as market order (was creating invalid $0.00 limit orders).
- **Unsettled Funds Position Sizing Fix**: Caps `sizing_base`, `buying_power`, and `options_buying_power` to `settled_cash` when available, preventing Good Faith Violations.
- **Session-Aware REST Price Guard**: REST fallback methods reject stale `last`/`lastPrice` values outside regular trading hours, accepting only bid/ask midpoint or Webull pPrice during extended hours.
- **SYNC-RISK Coordination Guard**: `broker_sync_service.py` checks the risk engine position cache before cancelling or closing trades, preventing erroneous actions due to transient broker API issues.
- **Channel Inheritance on Auto-Import**: `auto_import_manual_position` now inherits `channel_id` from OPEN/PENDING trades first (with full option identity matching including call_put), then falls back to recently-closed trades within 1hr. Risk engine pre-loads trade→position mappings at startup and refreshes every 30s, ensuring channel risk settings work even when Broker Sync Service is disabled.
- **Staleness Gate Selective Blocking**: Staleness gate no longer blanket-blocks all exits. It sets a `_staleness_is_blocking` flag and only blocks stop-loss type exits (SL, dynamic SL, trailing stop, giveback guard, EMA exits). Profit targets always pass through even on stale prices — selling at a stale high is favorable.
- **Stale Tier Flag Fix (Same-Symbol Re-Entry)**: Three-layer fix preventing stale `pt1_hit`/tier flags from carrying over when the same stock is re-entered on the same broker: (1) Zero-position cache cleanup runs after 3 consecutive cleanup windows with 0 positions and no active leases/pending orders/fill watches. (2) Trade-ID rollover detection in `get_or_create()` checks if the cached trade_id points to a CLOSED/CANCELLED trade — if so, full risk state reset regardless of closing/giveback flags. (3) Pending order fill confirmation now stores and validates `trade_id` to reject stale fills from a previous trade instance.
- **SPX Index Price Contamination Guard**: Routing engine price monitoring could misclassify options as equities when `position.strike` was 0/None, causing hub lookups to return the SPX index price (~$6,487) instead of the option premium (~$6.30). Fix: improved `is_option` detection using both `strike` and `option_type`, skip hub lookups for options with missing strike, and added a >50x entry-price ratio guard on both hub and REST prices to reject underlying index prices masquerading as option premiums.
- **Options Chain Multi-Broker Streaming Fix (5-part)**: Cross-subscription was sending Webull's numeric ticker IDs (e.g., `1059081524`) to Schwab's WebSocket which expects OCC symbols (`SPY   260401C00654000`). (1) Frontend now sends `expiry` in subscribe-stream request for OCC key construction. (2) Cross-sub builds proper OCC keys from `symbol.ljust(6) + YYMMDD + C/P + strike*1000 zfill(8)`. (3) `WebullDataHub.get_quote()` now accepts optional `max_age` parameter — chain overlay was failing with TypeError. (4) `SchwabDataHub` now has `_quotes_lock` for thread-safe access — stream-quotes endpoint was skipping Schwab entirely. (5) All three data paths (chain overlay, stream-quotes, SSE chain-stream) convert OCC→underscore format for frontend compatibility.
- **Advanced Tab Nesting Fix**: The `risk-tab-advanced` div in `channels.js` was nested INSIDE the `risk-tab-targets` div due to a missing `</div>` closing tag. When `switchRiskTab()` hid the targets pane, the advanced pane (as a child) was also hidden. Fixed by adding the missing closing div to make both tabs siblings. Also quoted all channel.id references in onclick handlers as strings to prevent JS precision loss on 18-19 digit Discord IDs.
- **Multi-Broker STC Qty Fix (3-part)**: (1) STC DB save now writes one trade per successful broker with actual `executed_qty` instead of a single shared entry using `signal['qty']`. (2) Added `original_quantity` column to trades table — set on BTO insert, preserved by sync when broker qty decreases after partial exits. (3) Initial trim qty query includes `original_quantity` for better estimates; TRIM FIX per-broker recalculation remains the authoritative correction.
- **Partial Exit PNL Fix (4-part)**: (1) STC trades now auto-link to BTO via `origin_trade_id` at save time (lookup open + closed BTO). (2) `reconcile_trade_fill_price` now calculates and sets PNL on individual STC trade records (not just BTO aggregate). (3) `broker_closed_position` in sync service aggregates PNL from all linked STCs using `original_quantity` instead of reduced qty. (4) Dashboard shows `original_quantity` for closed BTO trades with partial exits.
- **Daily PNL Baseline Fix**: `update_broker_pnl` was using the 9:30am SOD snapshot as baseline, which included unrealized gains from pre-market positions. When those positions closed for profit, the gains were invisible because they were already baked into the SOD baseline. Fix: always use the pre-market (4am) snapshot as baseline when available, capturing the full day's PNL. Also updates `sod_equity` in the locked-state branch so the corrected baseline propagates even when a broker is already locked.
- **IBKR/TastyTrade LEAPS Expiry Fix**: `_build_stc_signal()` converts option expiry from `YYYY-MM-DD` to `MM/DD` losing the year. `_direct_execute_exit` now reconstructs full `YYYY-MM-DD` expiry for IBKR and TastyTrade exits using the preserved `expiry_year` field, preventing wrong-year contract lookup on LEAPS.
- **TastyTrade Token Refresh on Order Placement**: `place_stock_order()` and `place_option_order()` now call `_ensure_session_valid()` before executing, preventing risk exit failures from expired 15-minute OAuth tokens.
- **Direct Exit OrderResult False-Positive Fix**: `_direct_execute_exit` previously converted any non-dict broker result to `str(result)` as order_id — `OrderResult(success=False)` would still appear as "success". Now checks `result.success` attribute: failed results log the failure message and skip success recording, triggering proper retry logic.
- **Conditional Order Multi-Broker Duplicate Fix**: When a channel had multiple brokers enabled (e.g., Webull + Schwab), the system created one conditional order per broker, causing duplicate executions on the same signal. Fix: all conditional order creation paths now use `brokers[0]` as primary and store the full broker list via `_all_brokers` in `parsed_signal`, persisted to the DB `metadata` JSON column. On trigger, the execution callback reads `all_brokers` from metadata and sets `_enabled_brokers` on the signal for multi-broker worker routing. This survives restarts (DB-persisted, not in-memory). All 10+ creation paths fixed: mapped/non-mapped COND ORDER, Hengy, Equity-Genie, registry, generic conditional, entry confirmation, India conditional, Telegram bridge, and edit/recreate (inherits from old order metadata).
- **Webull Bracket Order Client Fix**: `position_monitor.py` Webull bracket code referenced `broker_instance.wb` (from `src/brokers/webull_broker.py`) but the actual broker passed is the selfbot `WebullBroker` class which uses `_client`. Fixed all 3 call sites (initial SL, next PT, stop sync) to resolve the client via `getattr(broker_instance, '_client', None) or getattr(broker_instance, 'wb', None)` with null-safety guards. Also added price rounding consistent with `place_stock_order()`.
- **Broker Closure Exit Reason Fix**: `broker_sync_service.py` partial-exit and full-exit lot signal dicts were missing `exit_reason`, causing `lot_matcher.py` to default to `'MANUAL'`. Both paths now set `'exit_reason': 'BROKER_CLOSED'` for accurate exit classification in P&L tracking.
- **Conditional SL Percent-as-Fixed Auto-Correction**: When the signal parser reads "SL 10" (without `%` sign) from a conditional order signal, it stores `stop_loss_type='fixed'` and `stop_loss_fixed=10.0`. For penny stocks (e.g., AIXI at $0.38), a $10 fixed SL is above the entry price and triggers immediately. Fix: `base.py` `create_order` now detects when a "fixed" SL value exceeds the trigger price (nonsensical for a long entry) and auto-corrects it to `stop_loss_type='percent'` with `stop_loss_pct=value`, `stop_loss_fixed=None`. Logged as "SL auto-correct" for audit trail.
- **Exit Notification Classification Fix**: Notification dispatch in `position_monitor.py` was using fragile reason-string matching (`'STOP LOSS' in reason`, `'TRAILING' in reason`) to classify exit types. Dynamic SL exits showed as "stop loss hit", EMA exits sent no notification at all, and early trailing exits were missed (reason "EARLY TRAIL" doesn't contain "TRAILING"). Fix: dispatch now keys off `decision.risk_trigger` field (`stop_loss`, `dynamic_sl`, `trailing_stop`, `early_trailing`, `profit_target`, `giveback_guard`, `ema_exit`, `ema_no_trend`). Added `ExitDecision.dynamic_sl()` factory, `notify_dynamic_sl_triggered()` and `notify_ema_exit_triggered()` notification functions, and proper event recording for `DYNAMIC_SL`, `EMA_EXIT`, `EMA_NO_TREND` event types.
- **P&L Display Fix (bto_qty & Bad Tick Protection)**: `get_bot_trades()` in `gui_app/database.py` used `max(original_quantity, quantity)` for position sizing, which inflated positions when broker sync overwrote `quantity` with total broker position count (e.g., 2 BTO orders of 3 shares each → sync set both to qty=6, showing 6-share positions instead of 3). Fix: uses `original_quantity` when available (>0), falling back to `quantity`. Also added price sanity checks: (1) lot_closure display skips entries where exit_price/entry_price < 0.05 (catches bad ticks like $0.09 on a $2.69 stock), (2) `close_lot()` rejects suspicious close prices before persisting to prevent future bad-tick lot_closures.
- **Candle-Close SL Gap Protection**: On low-liquidity stocks (e.g., AIXI), the streaming price can lag behind the actual candle close. When a stock gaps from -8% to -22% in one candle, the streaming-based SL check sees -8% (no trigger), but the EMA engine sees the candle close at -22% and fires an EMA exit — bypassing the intended 10% SL. Fix: added a candle-close cross-check in `position_monitor.py` that runs BEFORE the EMA/enhanced risk evaluation. If `ema_risk_enabled` and the latest completed candle's close price shows a loss beyond `stop_loss_pct`, the SL triggers immediately using the candle price, ensuring the SL catches price gaps that streaming misses. Respects staleness gate and repair cycle guards. **Candle freshness guard**: only triggers if `last_candle_time` is within 3x the EMA timeframe — prevents false SL triggers from stale historical candles loaded during EMA pre-seeding (e.g., `seed_from_candles` loading old candles where close was much lower than current price).
- **IBKR Auto-Reconnect & Disconnect Detection**: `IBKRDataHub` now handles TWS/Gateway disconnection and automatic reconnection. Three-layer detection: (1) `disconnectedEvent` handler detects explicit TWS disconnects and marks streaming inactive, (2) TWS error codes 1100/1101/1102 handle connectivity lost/restored events (1101 triggers re-subscription of all market data symbols), (3) Reconciliation loop (10s interval) detects both explicit disconnects (`isConnected()=False`) and "zombie" connections (connected but no price updates for 90s+ across 3 consecutive checks). Auto-reconnect creates a fresh `IB()` instance, reconnects with stored host/port/clientId, swaps the broker's `ib` reference, re-attaches all event handlers, and re-subscribes `reqMktData` for all previously tracked symbols. 30s cooldown between reconnect attempts. Error codes 502 (can't connect) and 504 (not connected) are now logged for visibility. This fixes the issue where IBKR conditional order price monitoring would silently stop updating, requiring a manual "Connect" click in admin settings to restore.
- **Trading212 Extended Hours Support**: T212 API market orders placed after regular hours stay PENDING until next market open unless `extendedHours: true` is included in the request body. This is a T212 API-documented parameter (only supported on Market orders, not Limit/Stop/Stop-Limit). Fix: `trading212_client.py` `place_market_order()` now accepts `extended_hours` parameter, `trading212_broker.py` reads the `trading212_extended_hours` setting from database via `_get_extended_hours_enabled()` and passes it to the client. T212 added to the `supported_brokers` list in the extended hours API endpoint. Note: extended hours availability is instrument-specific on T212 — not all tickers support it.
- **ELPW Risk Monitor Bug Fix (Position-Trade Race Condition)**: When a position appeared in broker API before the corresponding trade was marked OPEN in the database, the risk engine detected it as "new" but couldn't match it to a trade (still PENDING). With auto-import disabled, the position was skipped as "external." On subsequent cycles, the position was no longer "new" (already in `_prev_position_keys`), so even after the trade became OPEN, no detection log fired. Two-part fix: (1) **Late trade match detection** — `_pending_trade_match_keys` tracks positions skipped due to no matching trade. When a previously-unmatched position finally gets a DB trade match (trade transitions to OPEN), the risk engine logs `TRADE MATCHED` and activates risk monitoring. `_skipped_external_logged` is cleared for re-logging. (2) **Position removal debounce (5s)** — brief 1-cycle position disappearances from Webull API no longer trigger `Position closed externally`. The key stays in `_prev_position_keys` during debounce, preventing false removal/re-detection cycles. Stale debounce entries are cleaned up when positions reappear or are confirmed gone.
- **UI Risk States Bug Fix (Dashboard Badges Not Showing)**: Two-part fix. (1) `get_all_risk_states()` in `position_cache.py` now computes SL/PT prices from channel settings percentages when explicit prices aren't set on the cache entry. During normal evaluation, SL/PT are computed on-the-fly from channel settings but never stored on the cache entry, so the UI always saw `stop_loss_active: false`. Now derives prices from entry_price × channel pct. Also added PT4 tier progression and aligned `stop_loss_active` with effective SL (dynamic/early/base). (2) `/api/risk-status` route and `get_position_cache()` now try both `src.risk.position_monitor` and `risk.position_monitor` in `sys.modules` to find the RiskManager instance, handling the dual-module namespace issue from mixed import styles. Added `channel_name`, `exit_mode`, `stop_loss_pct`, and `profit_target_pcts` to the API response for richer UI tooltips.
- **Schwab Position Fetch Fix (Risk Engine Blind Spot)**: `_fetch_schwab_cached()` in `position_monitor.py` had a logic bug where when `_schwab_streaming` was `True` but the Schwab data hub returned `None` for positions (hub position cache was never populated), the code would short-circuit to return `[]` at line 2915 — **never reaching the REST fallback**. This meant all Schwab positions were invisible to the risk engine: no SL/bracket monitoring, no exit triggers. Fix: changed the streaming cache guard to only return cached positions within `_REST_CACHE_TTL` (10s) instead of an unlimited 120s window, allowing the REST fallback to execute when the cache is stale or empty. The Schwab data hub's `get_positions()` never had position data because `update_positions(detailed=True)` was never called by the broker sync — only quote/streaming data was stored. With this fix, the risk engine falls through to `_fetch_schwab_positions()` (REST) which properly fetches from the Schwab broker API.

- **Order Chaser Cross-Broker Hub Priority Fix**: `_check_streaming_hubs_stock()` in `unfilled_order_chaser.py` previously checked all hubs in fixed order (Webull→Schwab→IBKR→TastyTrade) and returned the first with ANY data (`bid > 0 OR last > 0`). For Webull positions after-hours, Webull hub had `last > 0` (stale close price) with `bid=0, ask=0`, so the chaser used the stale price without ever checking Schwab's hub which had real bid/ask. Fix: (1) accepts `order_broker` parameter to identify the order's own broker, (2) two-pass priority — first seeks cross-broker hubs with real bid/ask (immediate return), then falls back to own-broker bid/ask, then any hub with last-only. This mirrors the risk engine's `_try_cross_hub_price()` which already skipped the position's own broker. Also passes `order.broker_id` from both call sites in `_get_exit_bid_ask`. Combined with the >10% price guard, this provides defense-in-depth against stale after-hours quotes replacing valid PT orders.
- **Webull Stream `pPrice` Priority Fix**: `webull_streaming_client.py` `close` field was overwriting `last` in the hub with the day-close price when no `pPrice` (after-hours price) was present. Fix: `close` now uses `setdefault` so it only sets `last` if no `pPrice` or `deal` trade has been received. This prevents the hub from serving stale close prices during after-hours when `pPrice` should take priority.
- **Webull Bracket Order Persistence & PT1 Fix (3-part)**: (1) `_place_initial_broker_bracket` in `position_monitor.py` prematurely set `cache.broker_orders_placed=True` in the Webull SL failure path (after-hours rejection), before the PT1 code could run. Removed the premature `True` — now only set at the end if at least one order was actually placed. (3) Webull PT1 stock order response used `pt_resp.get('orderId', '')` (top-level) but Webull returns `{'success': True, 'data': {'orderId': '...'}}` (nested). (4) `_get_broker_instance_for_bracket` returning `None` (broker not yet initialized on startup) previously silently set `broker_orders_placed=True` and returned — now logs warning and returns without setting flag, allowing retry.

- **Options Chain PUT Zigzag Root Cause Fix**: The Webull chain API returns flat rows (each with a `direction` field of `'call'` or `'put'`), NOT nested `{call: {...}, put: {...}}` dicts. `extract_option(row, direction)` in `selfbot_webull.py` checked `row.get(direction)` — got `None` (no nested key) — fell to the else branch which read row-level `tickerId`/`delta` for BOTH call and put directions. Result: every call row produced a duplicate "put" entry with the call's option_id/delta, and vice versa. Fix: in the flat-row fallback branch, check `row.get('direction')` and return `None` when it doesn't match the requested direction. Also added `bidList`/`askList` extraction in the flat branch (Webull uses price arrays, not scalar bid/ask). Additional UI fixes: (1) Round strikes to 2dp, dedup maps by highest bid. (2) Strike count selector (5/9/13/17/21/31 options).
- **SPX Hub Index Price Contamination Fix (Risk Engine)**: `_update_prices_from_hub()` was leaking the underlying index price into option positions via symbol fallback. For Webull options, the hub lookup tried `[pos.raw_symbol, pos.symbol]` — when the option's tickerId wasn't in the hub, it fell back to `pos.symbol` (e.g., "SPX") which matched the **underlying SPX index** quote (~$6900) instead of the option premium (~$5-50). This caused the risk engine to show options as e.g., +22% when they weren't. Three-layer fix: (1) Webull option hub lookup now uses ONLY `raw_symbol` (tickerId), never `pos.symbol` — prevents symbol-level index price from masquerading as option premium. (2) Added 50x ratio guard to ALL hub overlay paths (Webull, Schwab, IBKR, TastyTrade) — if hub price exceeds 50x the entry price, it's rejected as likely underlying index contamination, with throttled warning log. (3) Same 50x guard added to `_try_cross_hub_price()` for cross-broker option price lookups. The existing guard in `_resolve_webull_option_price()` only covered the REST fetch path; these new guards cover streaming hub overlay and cross-hub repair paths.
