# BotifyTrades — Compressed Documentation

## Overview

BotifyTrades is a production-grade, cross-platform automated trading bot designed to monitor Discord and Telegram for trade signals and execute them across multiple brokers for US and Canadian markets. It features a comprehensive Flask-based web control panel with real-time dashboards, advanced risk management, WebSocket streaming, AI-powered analysis, and detailed portfolio analytics. The project aims to provide automated trading capabilities, offering users a powerful tool for managing and optimizing their trading strategies.

## User Preferences

- **Security**: Always use environment variables (Replit Secrets) for credentials and license keys
- **Testing**: Test with paper_trade = true before enabling live trading
- **Monitoring**: Review console logs regularly for trade execution
- **Channel filtering**: Only process signals from designated channels
- **Deployment**: Prefer local machine or cloud VPS for 24/7 operation
- **Licensing**: All deployments require a valid license key (set via LICENSE_KEY environment variable or setup wizard)
- **Authentication**: First-time users are guided through setup wizard to create admin account with email recovery

## System Architecture

**UI/UX Decisions:**
The web control panel, built with Flask, provides real-time dashboards for broker status, live positions, P&L, and risk statuses. It includes detailed trade monitoring, performance analytics, intuitive settings, and visual feedback mechanisms like streaming indicators.

**Technical Implementations:**
- **Signal Sources**: Integrates with Discord and Telegram, utilizing a 5-tier parsing system (dedicated parsers, regex, AI fallback) for embed parsing, regex matching, and AI-powered detection, with deduplication and follow-up SL/PT updates. Channel Extract button fetches last 1000 raw messages as a downloadable JSON file for offline format analysis (no in-app pattern recognition or learned patterns UI).
- **Risk Management Engine**: A multi-layered system with global and per-channel settings, configurable Stop Loss, Trailing Stop, four Profit Targets, Dynamic SL Escalation, Max Profit Giveback Guard, EMA-5 Candlestick Risk Engine, and a "Leave Runner" feature. Risk states are persistent, and exit pricing prioritizes streaming hub bid/ask. **SL Escalation Only mode**: per-channel toggle where profit targets only escalate the stop loss (marking tier hits for dynamic SL) without triggering partial sells; closing is handled by Early Trailing + Giveback Guard.
- **Position Sizing**: Per-channel percentage-based sizing uses live broker data for options and stocks, considering specific buying power metrics across brokers. Three balance modes: "Live" (real-time BP), "Pre-Market" (4:00 AM ET snapshot), and "Start-of-Day" (9:30 AM ET snapshot). The SOD balance scheduler captures pre-market snapshots at 4 AM ET and market-open snapshots at 9:30 AM ET, with smart startup capture based on current time. **Conditional Order Sizing Mode**: `execute_conditional_order()` reads channel `sizing_mode` and sets `signal['_sizing_mode']` before any position sizing branch, so SOD/pre-market snapshots are used for percentage-based sizing in conditional orders (previously defaulted to live). Import of `get_channel_by_discord_id` moved to top of function to avoid `UnboundLocalError` from later local import shadowing.
- **Order Execution System**: An asynchronous, queue-based system for multi-broker order execution, featuring per-channel broker selection and a Universal Order Placement Resilience Layer with error classification, circuit breakers, and retries. Includes hub-first slippage checks and an Order Chaser service. Both Schwab and Webull brokers implement CBOE penny increment compliance ($0.05 under $3, $0.10 at/over $3; round DOWN for sells, UP for buys) on all STC option limit prices including retry loop prices. Schwab has non-blocking background fill verification — order returns SUCCESS immediately after HTTP 201 acceptance, a background task polls status (4 checks at 1s intervals) and auto-cancels DB trades if exchange rejects/cancels. Schwab API rate limit minimum interval set to 0.5s; httpx timeouts tightened from 15s/20s to 8s/10s. Background fill verification and post-BTO P&L refresh both skip when in 429 backoff (defers to sync service). Post-BTO P&L refresh delayed 8s to stagger behind background verify and avoid API contention.
- **Order Management System (OMS)**: Handles dynamic SL/PT management, exit order arbitration, position matching, and FIFO-based P&L tracking, supporting 12 types of exit source classifications.
- **WebSocket Streaming**: Utilizes Webull MQTT and an optimized Schwab WebSocket system for real-time quotes and orders, providing sub-100ms pricing for conditional orders and quick trade options chains, with automatic REST fallback. Schwab streaming auto-subscribes to all position symbols after WebSocket connect. Option ticker_ids are auto-subscribed on BTO order placement and slippage hub-miss to eliminate repeat hub misses for SPXW/SPX options.
- **Daily P&L Limit System**: Per-broker daily P&L tracking against SOD equity snapshots, with configurable dollar and percentage limits for profit targets and loss limits. Locks brokers from new BTO entries when thresholds are hit and resets at configured reset time. Snapshot type is dynamically resolved from `daily_pnl_reset_time` setting via `_resolve_snapshot_type()` — reset times before 9:30 AM use `pre_market` (4 AM) snapshot, 9:30+ uses `start_of_day` snapshot. Real-time P&L refresh on Webull order fills via MQTT streaming and immediate post-BTO account refresh for all brokers (via `asyncio.get_running_loop().create_task()` — runs in same event loop, no thread/loop conflicts). Dashboard polls every 15s.
- **Broker Sync Service**: Reconciles database state with actual broker states, detecting order fills/cancellations, position changes, and account updates, also feeding daily P&L limit service. Features mid-sync order interrupt (breaks out of sync loop when an order arrives), pre-queue sync pause (pauses sync at signal parse time before queue insertion), and async yield points between sync phases to prevent event loop starvation during order execution. Account info REST calls are throttled to once per 60s per broker (cached between cycles) to reduce API waste. **Pending Order Cancellation Safety**: (1) Grace period 120s (was 60s); (2) Consecutive empty-result guard requires 2 consecutive empty broker responses before processing PENDING trade cancellations; (3) Schwab re-verify via `get_order_status()` before cancellation — if order is still WORKING/PENDING, skips cancel; if FILLED, auto-transitions to OPEN; (4) Webull re-verify invalidates hub cache and re-fetches fresh pending orders + checks order history before cancellation.
- **Order Chaser Service**: Monitors unfilled exit/entry orders and replaces stale ones with better prices. DST-aware market hours gate prevents chasing/cancelling off-hours (weekends, before 4 AM ET, after 8 PM ET). False-fill detection verifies order status via `get_order_status()` and position lookup before marking orders as filled — prevents Schwab false-fills when orders are rejected/cancelled by broker. Position-based verification works for both BTO (symbol found = filled) and STC (symbol still present = NOT filled). UNKNOWN status treated as cancelled (safe default) for both entry and exit orders — never falsely marks unfilled orders as filled. `[ORDER_CHASER]` prefix whitelisted in `CleanConsoleFormatter.CONSOLE_WHITELIST` for console visibility.
- **Position Cache Flip-Flop Guard**: Entry price updates include a flip-flop detector that tracks the last 3 entry prices per position key. When the broker alternates between two values (e.g., two SPX positions with different avg_costs mapping to the same key), the entry price is locked to the first value seen to prevent risk engine oscillation.
- **Broker Health Monitor**: Creates `broker_states` and `broker_notifications` tables on init. Exposes `get_broker_state()` method for GUI compatibility.
- **Schwab Token Auto-Refresh**: `SchwabTokenManager` singleton runs a daemon thread refreshing access tokens 5 min before expiry (30-min lifetime). Refresh tokens last 7 days (Schwab limitation — re-auth required weekly). After GUI re-authentication, `_hot_connect_schwab_broker()` immediately connects the broker (~1s), updates BrokerManager + health monitor + sync service. Console log prefixes `[SCHWAB TOKEN]` and `[SCHWAB HOT-CONNECT]` are whitelisted in `logging_config.py`.
- **Sub-1s Order Execution Optimizations**: (1) **Parse-time option_id prefetch** — `_prefetch_option_id()` awaited before queue.put() eliminates 4-6s REST lookup from execution path by pre-caching option_id in `_option_id_cache`; guarded by `hasattr(self, '_prefetch_option_id')` for compatibility with modular SelfClient; skips index symbols (SPXW/SPX/NDX/VIX etc.) that aren't on Webull; has 10s timeout to prevent indefinite hangs; (2) **30s prewarm** — `_prewarm_scheduler()` starts 30s after boot (was 5 min), covers top symbols + last 7 days DB; (3) **Slippage REST elimination** — when `allow_when_no_quote=true` AND hub misses, skips REST quote entirely (limit price protects); (4) **FAST-PATH timing** — `_parsed_at`/`_queued_at` timestamps + `[FAST-PATH]` log showing Parse→Queue/Queue→Worker/Health/Setup/API/TOTAL breakdown; (5) Cache-first position sizing, channel info DB cache with 10s TTL, bounded sync_ready gate, signal price fallback to all brokers.
- **Notification System**: Provides Discord webhook and desktop browser notifications for trading events.
- **Security & Authentication**: Implements admin account management, password hashing, email recovery, session-based authentication, rate limiting, and encrypted broker credentials. Schwab OAuth uses CSRF state tokens, PKCE, and a postMessage-based popup flow.
- **Database Architecture**: Uses SQLite with WAL mode for concurrent read/write operations, storing critical trading data and encrypted credentials. Position ledger uses `_execute_with_retry()` with exponential backoff (3 retries) on `OperationalError: database locked` for critical exit bookkeeping. Ledger UNIQUE index includes `routing_mapping_id` to prevent position collision across routing mappings.
- **AI Analysis**: Integrates OpenAI GPT for pre-trade and post-trade analysis, an AI chat assistant, and AI command toggles.
- **Risk Engine Direct Exit Architecture**: Dual-path exit execution system ensuring stop-loss orders execute even when the event loop is blocked, using a primary queue-based path and a backup daemon thread for direct broker calls. Exit retry mechanism clears `_exit_executed_keys` on failure so subsequent retries are not blocked by stale dedupe state. Backup thread distinguishes "worker succeeded" from "worker failed and reset closing flag" by checking `entry.exit_retry_count > 0` — prevents false "worker handled it" logs after failed STC. **Exit Lease Manager** (`src/risk/exit_lease_manager.py`): Thread-safe per-position exit lease with states (idle/queued/executing), owner tokens (risk_engine/worker/chaser/backup_thread), conditional ownership transfer (`expected_owner` CAS semantics), and 60s auto-expiry. Prevents duplicate exits across all pipelines. `_exit_executed_keys` kept as secondary dedup guard. **Exchange Rejection Recovery**: When Schwab API accepts an order (200+order_id) but the exchange rejects it asynchronously, both `_background_verify_order()` (schwab_broker.py) and the order chaser (unfilled_order_chaser.py) now clear `_exit_executed_keys` and call `cache.record_exit_failure(pos_key, reason, is_stop_loss=True)` — enabling market order escalation on retry. Previously, only the "All brokers failed" worker path cleared the dedup key, causing exchange-rejected STC orders to be silently blocked indefinitely.
- **Risk Engine Speed Optimizations**: Interval floor lowered from 1.0s to 0.2s (configurable 0.2–60s range); early-wake sleep chunk 0.1s; `_check_service_enabled()` cached with 10s TTL; `cache.save()` throttled to every 2s. Closing flag uses wall-clock time (`closing_since` field via `time.monotonic()`) instead of cycle count, making timeout independent of loop speed. **Parallel Broker Fetches**: `_fetch_all_positions()` runs ALL broker fetch tasks concurrently via `asyncio.gather` — total cycle time = max(single_broker) instead of sum(all_brokers). Each broker fetch is wrapped in an async closure with its own cache/rate_manager logic. **REST Position Cache (10s TTL)**: Per-broker REST cache reuses results within 10 seconds. Force-bypass via `_force_rest_refresh` flag set on early-wake order events. **SchwabDataHub Hub-First**: Schwab positions fetched directly from `SchwabDataHub.get_positions(detailed=True)` before falling back to REST — zero API cost when hub has data. **Cross-Broker Price Updates**: `_update_prices_from_hub()` updates non-streaming broker positions (Alpaca, IBKR, Tastytrade, Robinhood) with real-time prices from Webull/Schwab streaming hubs for stock symbols — ensures price freshness between REST cache refreshes. Cycle timing logged as `[RISK] ⏱ Cycle #N: Xms` (first 5, every 60th, and always when >2000ms).
- **Streaming Hub-First Architecture**: Comprehensive hub-first + REST fallback pattern across all services. Both `get_webull_data_hub()` and `get_schwab_data_hub()` are thread-safe singletons (double-checked locking). WebullDataHub caches account info (90s TTL), positions (45s TTL), pending orders (45s TTL), ticker IDs, option IDs (600s TTL), and streaming quotes (120s stale threshold). Risk engine reads positions directly from WebullDataHub (zero API cost, up to 60s stale accepted) and NEVER goes blind due to rate limiting — always falls back to in-memory cached snapshots. Rate limit budget: 30 requests/60s for Webull (reduced from unlimited competing consumers). Streaming client debounces order-event REST refreshes (5s cooldown). Sync service staggers account info fetches (every 3rd cycle). Position fetcher extracts option metadata from position data directly instead of making per-option REST API calls. **All Webull broker methods** (`get_account_info`, `get_pending_orders`, `get_order_history`, `get_positions_detailed`) use hub cache with REST fallback — results stored back to hub on REST fetch; cache invalidated on order placement/cancel. `refresh_orders_once()` normalizes raw Webull order data (orderId→order_id, totalQuantity→quantity etc.) before caching. Sync service skips filled order history when broker has 0 positions and 0 DB trades. Order history cached 120s in broker. Services covered: selfbot_webull.py, routes.py, broker_live_analytics.py, price_monitor_service.py, trade_tracker.py, ndx_qqq_converter.py, webull_broker.py, position_monitor.py. **SchwabDataHub** mirrors the same pattern: caches account info (30s TTL), pending orders (15s TTL), positions (15s TTL), order history (60s TTL), streaming quotes. All Schwab broker methods (`get_account_info`, `get_pending_orders`, `get_order_history`, `get_positions_detailed`) use hub cache with REST fallback — results stored back to hub on REST fetch; cache invalidated on all order placement (stock, option, stop, stop-limit, trailing stop, OCO, bracket), cancel, and replace.

**System Design Choices:**
- **Broker Isolation**: Prevents data cross-contamination by isolating each broker's streaming data.
- **Thread Safety**: All shared states are protected by locks. `PositionCache._cache` uses `threading.RLock` for cross-thread safety (risk monitor + Flask UI).
- **Priority Queue Execution**: `_PriorityOrderQueue` wraps `asyncio.PriorityQueue` — risk management exits (priority 0) always dequeue before normal entries (priority 1), preventing stop-loss delays during high signal volume.
- **Hub-First Architecture**: Prioritizes cached streaming data over REST API calls.
- **Graceful Degradation**: Automatically falls back to REST polling if streaming services are unavailable.
- **Modular Broker Abstraction**: Uses a common interface for diverse broker APIs.
- **Market Isolation**: Conditional order services for different markets operate independently.
- **Conditional Order Guards**: Includes per-channel Breakout Reset Guard and Limit Cap.
- **Single-Application Trigger Offset**: Offset applied once at conditional order creation (stored as `adjusted_trigger_price`), never re-applied at execution time. `original_signal_price` column preserves the raw signal price for dynamic offset recalculation.
- **Webull Option Direction Normalization**: `get_positions_detailed()` in webull_broker.py and position_monitor.py hub path both accept `C/P/CALL/PUT` (case-insensitive) for option direction. Prevents SPX-style options from being misclassified as stocks when Webull returns short-form direction codes.
- **Reverse Option ID Cache**: `webull_broker.py` maintains `_reverse_option_id_cache` mapping `optionId → {symbol, strike, expiry, option_type}`. Populated in `cache_option_id()`, used by `get_option_details_by_id()` for reverse lookup when Webull returns strike=0 for index options (SPX/NDX). Fallback chain: optionId → tickerId → regex parsing. Used in `get_positions_detailed()`, `live_snapshot.py`, and `position_monitor.py`. **Selfbot path** (`selfbot_webull.py`) uses inline reverse scan of `_option_id_cache` (no separate reverse cache) for the same enrichment.
- **Sync Pre-Enrichment (Stage 0)**: `broker_sync_service.py` `_pre_enrich_option_positions()` runs before Stage 1 (`_update_existing_trades`), recovering missing strike/expiry/call_put from DB OPEN/PENDING BTO trades. Prevents Stage 1 key mismatch → false cancel of tracked trades (root cause of the SPY_stock incident).
- **Risk Engine Safe STC Construction**: `_build_stc_signal()` returns `None` instead of defaulting to opt_type='C' when direction is unknown. Tries DB trade record lookup via `get_trade_by_id()` first. Caller in `_handle_exit_decision` cleanly releases exit lease on `None` return.
- **Schwab Fresh Quote for Orders**: `get_option_quote()` accepts `max_age` parameter (seconds) threaded through `get_hub_quote_detailed()` → `SchwabDataHub.get_quote(max_age=)`. Order placement uses `max_age=10` to avoid stale 120s hub cache for pricing.
- **STC option_id Copy-Through**: STC pre-check in selfbot_webull.py copies `option_id` and `expiry_full` from the matched broker position into the signal dict, ensuring Webull `place_option_order()` has the required option_id for execution.
- **Order Chaser Replacement Orders**: Uses `option_type=` (not `direction=`) kwarg when calling `place_option_order()` for replacement orders, matching the standard broker interface signature.
- **Schwab Streaming Subscribe**: `subscribe_position_symbols()` checks both `assetType` and `asset` fields with values `OPTION`/`option`, ensuring normalized positions get subscribed for streaming quotes.
- **Slippage Enforcement Deferral**: Stage A (base.py) logs slippage warnings but does not kill orders; enforcement is deferred to Stage B/C broker pipeline wait-and-recover.
- **Entry Confirmation Independence**: `_entry_confirmation` flag bypasses `conditional_order_enabled` channel setting to create conditional orders from entry confirmation signals.
- **Dynamic Offset Recalculation**: `update_conditional_order_trigger_offset()` uses `original_signal_price` as baseline, supports both percent and dollar modes, and recalculates `limit_price` when limit cap is enabled.
