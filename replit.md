# BotifyTrades — Compressed Documentation

## Overview

BotifyTrades is a production-grade, cross-platform automated trading bot designed to monitor Discord and Telegram for trade signals and execute them across multiple brokers for US and Canadian markets. It features a comprehensive Flask-based web control panel with real-time dashboards, advanced risk management, WebSocket streaming, AI-powered analysis, and detailed portfolio analytics. The project aims to provide automated trading capabilities, offering users a powerful tool for managing and optimizing their trading strategies.

## User Preferences

- **Security**: Always use environment variables (Replit Secrets) for credentials and license keys
- **Testing**: Test with paper_trade = true before enabling live trading
- **Monitoring**: Review console logs regularly for trade execution
- **Channel filtering**: Only process signals from designated channels
- **Deployment**: Prefer local machine or cloud VPS for 24/7 operation
- **Licensing**: All deployments require a valid license key (set via LICENSE_KEY environment variable or setup wizard)
- **Authentication**: First-time users are guided through setup wizard to create admin account with email recovery

## System Architecture

**UI/UX Decisions:**
The web control panel, built with Flask, provides real-time dashboards for broker status, live positions, P&L, and risk statuses. It includes detailed trade monitoring, performance analytics, intuitive settings, and visual feedback mechanisms like streaming indicators.

**Technical Implementations:**
- **Signal Sources**: Integrates with Discord and Telegram, utilizing a 5-tier parsing system (dedicated parsers, regex, AI fallback) for embed parsing, regex matching, and AI-powered detection, with deduplication and follow-up SL/PT updates. Channel Extract button fetches last 1000 raw messages as a downloadable JSON file for offline format analysis (no in-app pattern recognition or learned patterns UI).
- **Risk Management Engine**: A multi-layered system with global and per-channel settings, configurable Stop Loss, Trailing Stop, four Profit Targets, Dynamic SL Escalation, Max Profit Giveback Guard, EMA-5 Candlestick Risk Engine, and a "Leave Runner" feature. Risk states are persistent, and exit pricing prioritizes streaming hub bid/ask. **SL Escalation Only mode**: per-channel toggle where profit targets only escalate the stop loss (marking tier hits for dynamic SL) without triggering partial sells; closing is handled by Early Trailing + Giveback Guard. **EMA in Signal Mode**: When `exit_strategy_mode='signal'` but `ema_risk_enabled=true`, the risk engine bypasses the standard signal-mode early-return and still evaluates EMA risk (via `_evaluate_enhanced_risk`), executing EMA-triggered exits with market hours checking while skipping all other automated risk management (SL, trailing, profit targets). Previously, `evaluate_exit_actions()` in `risk_engine.py` had a redundant `exit_strategy_mode != 'signal'` gate on the EMA block (section 2.5) that contradicted the outer signal-mode EMA path in `position_monitor.py` — this made the entire signal-mode EMA path dead code. Fixed by removing the inner gate; the outer caller already controls scope. **EMA Singleton via builtins**: `get_candle_service()` stores the `CandlePreWarmService` singleton in `builtins._ema_candle_service_singleton` to prevent dual-instance bugs when the module is imported via different Python paths (absolute `src.risk.ema_engine` vs relative `.ema_engine`).
- **Position Sizing**: Per-channel percentage-based sizing uses live broker data for options and stocks, considering specific buying power metrics across brokers. Three balance modes: "Live" (real-time BP), "Pre-Market" (4:00 AM ET snapshot), and "Start-of-Day" (9:30 AM ET snapshot). The SOD balance scheduler captures pre-market snapshots at 4 AM ET and market-open snapshots at 9:30 AM ET, with smart startup capture based on current time. **Conditional Order Sizing Mode**: `execute_conditional_order()` reads channel `sizing_mode` and sets `signal['_sizing_mode']` before any position sizing branch, so SOD/pre-market snapshots are used for percentage-based sizing in conditional orders (previously defaulted to live). Import of `get_channel_by_discord_id` moved to top of function to avoid `UnboundLocalError` from later local import shadowing.
- **Order Execution System**: An asynchronous, queue-based system for multi-broker order execution, featuring per-channel broker selection and a Universal Order Placement Resilience Layer with error classification, circuit breakers, and retries. Includes hub-first slippage checks and an Order Chaser service. Both Schwab and Webull brokers implement CBOE penny increment compliance ($0.05 under $3, $0.10 at/over $3; round DOWN for sells, UP for buys) on all STC option limit prices including retry loop prices. Schwab has non-blocking background fill verification — order returns SUCCESS immediately after HTTP 201 acceptance, a background task polls status (4 checks at 1s intervals) and auto-cancels DB trades if exchange rejects/cancels. Schwab API rate limit minimum interval set to 0.5s; httpx timeouts tightened from 15s/20s to 8s/10s. Background fill verification and post-BTO P&L refresh both skip when in 429 backoff (defers to sync service). Post-BTO P&L refresh delayed 8s to stagger behind background verify and avoid API contention. **Schwab Bracket STC Conflict Resolution**: `_cancel_conflicting_sell_orders()` method in `schwab_broker.py` automatically cancels any WORKING sell orders (bracket SL/PT OCO legs) for a symbol before placing a new STC. This prevents Schwab from rejecting risk engine Dynamic SL exits when bracket order legs are still active. Called from both `place_stock_order()` (EQUITY) and `place_option_order()` (OPTION with exact OCC symbol match). Skipped in dry-run mode. Includes 300ms settle delay after cancellation. **Market Order Chase Fix**: Redesigned from aggressive 3-iteration loop (2.5s wait) to single-chase architecture (5s wait). Flow: wait 5s for fill → if unfilled, fetch new price at max buffer BEFORE cancelling → only cancel if valid replacement price obtained → if cancel succeeds, `_order_is_live=False` until replacement confirmed → on double replacement failure, result marked `error='CHASE_FAILED'` with `_chase_cancelled=True` so downstream/order chaser won't track dead orderId. OrderId extraction normalized to check both `result.orderId` and `result.data.orderId`. Order cache invalidated before fill-status check to prevent stale reads.
- **Order Management System (OMS)**: Handles dynamic SL/PT management, exit order arbitration, position matching, and FIFO-based P&L tracking, supporting 12 types of exit source classifications. **Universal Risk Exit P&L**: All broker risk exits (Webull, Tastytrade, Schwab, IBKR) now record Signal P&L (lot_closures via lot_matcher) and Execution P&L (execution_closures via record_execution_closure_atomic) immediately on successful STC order placement — not deferred to sync. Alpaca path already had this; non-Alpaca paths added in selfbot_webull.py post-execution success handler and Tastytrade early-return path. **Multi-Lot Overflow Closure**: `record_execution_closure_atomic()` now loops through all matching FIFO execution lots instead of `LIMIT 1`, properly distributing close quantity across lots when a single STC covers more shares than the first lot. Each lot gets its own closure record with correct per-lot P&L. Prevents orphaned OPEN execution lots when duplicate BTO trades create multiple lots for the same position. **Consolidated Fill-Based P&L**: Single unified P&L view replacing the previous two-tab (Signal/Execution) system. `signal_lots` stores `entry_fill_price`, `entry_fill_broker`, `entry_fill_order_id`, `entry_filled_at`; `lot_closures` stores `exit_fill_price`, `exit_fill_broker`, `exit_fill_order_id`, `exit_filled_at`, `exit_source`. P&L calculation uses fill price when available (from BrokerSyncService), falls back to signal price. API `/api/pnl/detailed` returns `has_fill_data`, `entry_fill_broker`, `signal_entry_price` alongside `entry_price` (best available). UI shows broker badges and "Fill"/"Signal" indicators on entry/exit prices. DB functions `update_lot_entry_fill()` and `update_closure_exit_fill()` recalculate P&L with fill-based entry/exit prices — both use **first-write-wins** semantics (skip update if fill already recorded, preventing multi-broker last-write-wins corruption). LotMatcher exposes `record_entry_fill()` and `record_exit_fill()` methods. BrokerSyncService calls these in `_record_execution_lot` and `_record_execution_closure` to pipe broker fills into signal lots. **Transaction Safety**: `close_lot()` uses `BEGIN IMMEDIATE` to prevent concurrent STC/risk actions from over-closing; also caps `close_qty` to `remaining_qty` and rejects already-closed lots. **Closure Matching**: BrokerSyncService matches broker exit fills to signal lot_closures using multi-factor ordering (quantity proximity, price proximity, then recency) instead of naive `closed_at DESC`. **Performance Indexes**: `idx_lot_closures_lot_id` and `idx_signal_lots_signal_id` added for JOIN performance. **Pagination**: `/api/pnl/detailed` supports `page` and `per_page` query params (default 100, max 500) with `total_pages`, `has_next`, `has_prev` metadata.
- **WebSocket Streaming**: Utilizes Webull MQTT and an optimized Schwab WebSocket system for real-time quotes and orders, providing sub-100ms pricing for conditional orders and quick trade options chains, with automatic REST fallback. Schwab streaming auto-subscribes to all position symbols after WebSocket connect. Option ticker_ids are auto-subscribed on BTO order placement and slippage hub-miss to eliminate repeat hub misses for SPXW/SPX options. **Cross-Broker Streaming Subscription**: Risk engine detects positions on non-streaming brokers (Alpaca, IBKR, Robinhood, Tastytrade) and requests the Schwab streaming client to subscribe their equity symbols via a thread-safe pending subscription queue in SchwabDataHub (`request_subscribe_equities` → `drain_pending_subscriptions`). The Schwab streaming client drains pending requests every 10s in its message loop. Failed subscriptions are re-queued automatically. This ensures all monitored positions get real-time streaming prices (~21ms tick-to-eval latency) regardless of which broker holds them. **Immediate BTO Subscribe**: Both Webull and Schwab `place_stock_order()` and `place_option_order()` now immediately subscribe newly opened symbols to their streaming clients on successful BTO (eliminates 30s sync-cycle gap). Webull uses hub ticker_id lookup; Schwab calls `subscribe_equities()`/`subscribe_options()`. **Webull subscribe_position_symbols()**: Added to WebullBroker (delegates to `_streaming_client.subscribe_positions()`) so `broker_sync_service.py` can auto-subscribe Webull positions every sync cycle (was silently skipped before due to missing method — root cause of ACXP stale price bug).
- **Daily P&L Limit System**: Per-broker daily P&L tracking against SOD equity snapshots, with configurable dollar and percentage limits for profit targets and loss limits. Locks brokers from new BTO entries when thresholds are hit and resets at configured reset time. Snapshot type is dynamically resolved from `daily_pnl_reset_time` setting via `_resolve_snapshot_type()` — reset times before 9:30 AM use `pre_market` (4 AM) snapshot, 9:30+ uses `start_of_day` snapshot. Real-time P&L refresh on Webull order fills via MQTT streaming and immediate post-BTO account refresh for all brokers (via `asyncio.get_running_loop().create_task()` — runs in same event loop, no thread/loop conflicts). Dashboard polls every 15s. **SOD Snapshot Restart Protection**: `capture_snapshot()` skips re-capture if today's snapshot already exists (unless `force=True`). Scheduled 4AM/9:30AM captures use `force=True`; startup captures don't. Snapshots are persisted to `sod_snapshots.json` and restored on startup, so mid-day restarts preserve the original baseline instead of resetting P&L to the current (post-loss) balance. **Max Daily Trades Per Broker**: Configurable trade count limit (`max_daily_trades_default` in `global_risk_settings`, per-broker overrides via `max_daily_trades_overrides` JSON). After N successful BTO submissions, broker gets `lock_type='trades'` blocking further BTOs. Only counts actual successful BTO placements (not STC, not chase replacements, not broker sync imports). Counter tracked in `daily_pnl_state.daily_trade_count`, preserved during P&L equity updates (DB upsert uses `CASE WHEN excluded.daily_trade_count IS NOT NULL`), resets at same daily reset time. Dashboard shows orange trade-limit banner with count/limit. Setting 0 = unlimited. `check_broker_locked()` has catch-up lock path for edge case where count reaches limit between record and check calls.
- **Broker Sync Service**: Reconciles database state with actual broker states, detecting order fills/cancellations, position changes, and account updates, also feeding daily P&L limit service. Features mid-sync order interrupt (breaks out of sync loop when an order arrives), pre-queue sync pause (pauses sync at signal parse time before queue insertion), and async yield points between sync phases to prevent event loop starvation during order execution. Account info REST calls are throttled to once per 60s per broker (cached between cycles) to reduce API waste. **Pending Order Cancellation Safety**: (1) Grace period 120s (was 60s); (2) Consecutive empty-result guard requires 2 consecutive empty broker responses before processing PENDING trade cancellations; (3) Schwab re-verify via `get_order_status()` before cancellation — if order is still WORKING/PENDING, skips cancel; if FILLED, auto-transitions to OPEN; captures `status_description` from Schwab API and stores in both `close_reason` and `rejection_reason` columns for post-mortem diagnosis of rejected SPX orders; (4) Webull re-verify invalidates hub cache and re-fetches fresh pending orders + checks order history before cancellation.
- **Order Chaser Service**: Monitors unfilled exit/entry orders and replaces stale ones with better prices. DST-aware market hours gate prevents chasing/cancelling off-hours (weekends, before 4 AM ET, after 8 PM ET). False-fill detection verifies order status via `get_order_status()` and position lookup before marking orders as filled — prevents Schwab false-fills when orders are rejected/cancelled by broker. Position-based verification works for both BTO (symbol found = filled) and STC (symbol still present = NOT filled). Exit UNKNOWN keeps order tracked as PENDING (safe retry). Entry UNKNOWN requires 3 consecutive unknown results before untracking (prevents premature removal during API outages). `[ORDER_CHASER]` prefix whitelisted in `CleanConsoleFormatter.CONSOLE_WHITELIST` for console visibility. **Risk Manager Wiring**: `BrokerManager._risk_manager` is set after RiskManager creation so chaser max-attempt/rejection recovery paths can call `record_exit_failure()` and clear `_exit_executed_keys`. **Price-First Cancel Pattern**: Exit chase fetches replacement price BEFORE cancelling the existing order — if no price available, existing order stays alive. **Replacement Retry**: Both entry and exit replacement failures trigger one immediate retry at same price; if retry also fails, exit path releases lease + records failure for risk engine retry, entry path untracks the dead order. Prevents stale cancelled order IDs from being tracked as PENDING. **Signal STC Chasing**: ALL STC orders are now tracked by the chaser (not just risk management exits). `TrackedExitOrder.is_risk_order` flag distinguishes risk STCs from signal STCs — signal STCs (`is_risk_order=False`) skip exit lease release and risk engine failure recording on failure/cancel, since they have no associated risk state. All three STC tracking sites (single-broker, multi-broker, Alpaca) pass `is_risk_order` explicitly. **Bracket Metadata Preservation**: `TrackedEntryOrder` carries `stop_loss_price` and `profit_target_price` fields. `_place_entry_replacement_order()` uses `place_bracket_order()` when bracket metadata is present, with explicit success check and fallback to simple `place_stock_order()` on failure. Both `track_entry_order` call sites extract bracket SL/PT from signal using canonical keys first (`stop_loss_price`, `profit_target_price`) then aliases. Bracket metadata is preserved through replacement order re-tracking.
- **Schwab Stock Exit Order Strategy**: Two-tier approach in `place_stock_order()`: (1) During **NORMAL session** (regular market hours 9:30-16:00 ET), stock STC exits with `_use_market_order=True` use **true MARKET orders** — no limit price conversion. This avoids Schwab's price-band rejection on OTC/low-volume stocks (e.g., SER at $2.15 was repeatedly rejected even with near-touch LIMIT at bid-1¢). (2) During **SEAMLESS session** (extended hours), falls back to aggressive LIMIT via `_get_aggressive_exit_price()` using near-touch pricing: `bid - 1 tick` (1¢ for ≥$1, 0.01¢ for sub-$1), clamped to last-trade band (2%/3%/5% tiers). Helper methods: `_stock_tick_below()`, `_clamp_to_last_band()`. Option exit pricing unchanged (still `bid × 0.95` with CBOE rounding).
- **Schwab OrderSessionPolicy**: `_get_session_type()` auto-detects premarket/after-hours via `_is_regular_market_hours()` (checks ET timezone, weekday, 9:30-16:00 window) and returns `SEAMLESS` without requiring DB extended_hours flag. Both `place_stock_order()` and `place_bracket_order()` use `GOOD_TILL_CANCEL` duration when session is `SEAMLESS` (previously hardcoded `DAY` causing premarket orders to never fill). DB extended_hours toggle still works as an explicit override. Webull `place_bracket_order()` entry orders also use `GTC` + `outsideRegularTradingHour=True` to match `place_stock_order()` behavior.
- **Position Cache Flip-Flop Guard**: Entry price updates include a flip-flop detector that tracks the last 3 entry prices per position key. When the broker alternates between two values (e.g., two SPX positions with different avg_costs mapping to the same key), the entry price is locked to the first value seen to prevent risk engine oscillation.
- **Broker Health Monitor**: Creates `broker_states` and `broker_notifications` tables on init. Exposes `get_broker_state()` method for GUI compatibility.
- **IBKR Dashboard Wiring Fix**: Three-layer fix ensuring IBKR (and all brokers) show on the dashboard even when disconnected: (1) `selfbot_webull.py` no longer nulls `self.ibkr_broker` on connection timeout/failure — the IBKRBroker instance stays alive with `connected=False` so the dashboard can reference it; `set_broker_status()` is called on failure with error reason. (2) Index route filter now includes brokers with an `error` field set (configured-but-disconnected brokers with a failure reason). (3) `/api/v2/broker-states` has a credentials-based fallback — when `broker_instance is None` but credentials exist in the DB, a disconnected state entry is returned showing "configured but not connected" with correct paper/live labeling derived from stored `paper_mode`.
- **Schwab Token Auto-Refresh**: `SchwabTokenManager` singleton runs a daemon thread refreshing access tokens 5 min before expiry (30-min lifetime). Refresh tokens last 7 days (Schwab limitation — re-auth required weekly). After GUI re-authentication, `_hot_connect_schwab_broker()` immediately connects the broker (~1s), updates BrokerManager + health monitor + sync service. Console log prefixes `[SCHWAB TOKEN]` and `[SCHWAB HOT-CONNECT]` are whitelisted in `logging_config.py`.
- **Sub-1s Order Execution Optimizations**: (1) **Parse-time option_id prefetch** — `_prefetch_option_id()` awaited before queue.put() eliminates 4-6s REST lookup from execution path by pre-caching option_id in `_option_id_cache`; guarded by `hasattr(self, '_prefetch_option_id')` for compatibility with modular SelfClient; skips index symbols (SPXW/SPX/NDX/VIX etc.) that aren't on Webull; has 10s timeout to prevent indefinite hangs; (2) **30s prewarm** — `_prewarm_scheduler()` starts 30s after boot (was 5 min), covers top symbols + last 7 days DB; (3) **Slippage REST elimination** — when `allow_when_no_quote=true` AND hub misses, skips REST quote entirely (limit price protects); (4) **FAST-PATH timing** — `_parsed_at`/`_queued_at` timestamps + `[FAST-PATH]` log showing Parse→Queue/Queue→Worker/Health/Setup/API/TOTAL breakdown; (5) Cache-first position sizing, channel info DB cache with 10s TTL, bounded sync_ready gate, signal price fallback to all brokers.
- **Notification System**: Provides Discord webhook and desktop browser notifications for trading events.
- **Security & Authentication**: Implements admin account management, password hashing, email recovery, session-based authentication, rate limiting, and encrypted broker credentials. Schwab OAuth uses CSRF state tokens, PKCE, and a postMessage-based popup flow.
- **Database Architecture**: Uses SQLite with WAL mode for concurrent read/write operations, storing critical trading data and encrypted credentials. Position ledger uses `_execute_with_retry()` with exponential backoff (3 retries) on `OperationalError: database locked` for critical exit bookkeeping. Ledger UNIQUE index includes `routing_mapping_id` to prevent position collision across routing mappings.
- **AI Analysis**: Integrates OpenAI GPT for pre-trade and post-trade analysis, an AI chat assistant, and AI command toggles.
- **Risk Engine Direct Exit Architecture**: Dual-path exit execution system ensuring stop-loss orders execute even when the event loop is blocked, using a primary queue-based path and a backup daemon thread for direct broker calls. Exit retry mechanism clears `_exit_executed_keys` on failure so subsequent retries are not blocked by stale dedupe state. Backup thread distinguishes "worker succeeded" from "worker failed and reset closing flag" by checking `entry.exit_retry_count > 0` — prevents false "worker handled it" logs after failed STC. **Exit Lease Manager** (`src/risk/exit_lease_manager.py`): Thread-safe per-position exit lease with states (idle/queued/executing), owner tokens (risk_engine/worker/chaser/backup_thread), conditional ownership transfer (`expected_owner` CAS semantics), and 180s auto-expiry (increased from 60s). Prevents duplicate exits across all pipelines. **Stop-Loss Spam Prevention**: Exit lease acquisition and `mark_closing` now execute BEFORE event recording and Discord notifications in `_execute_exit()`, preventing duplicate STOP_LOSS events/notifications from flooding on every risk cycle. `CLOSING_TIMEOUT_SECONDS` also 180s to match lease TTL. Arbiter rejection path releases lease+closing to prevent lockout. `_exit_executed_keys` kept as secondary dedup guard. **`release_exit_marker()` Helper**: Centralized cleanup method on `RiskManager` that atomically discards from `_exit_executed_keys` and force-releases the exit lease. Called from: (1) worker success path in selfbot_webull.py, (2) direct-exit backup thread (both success and exception branches), (3) 61s closing timeout reset in position_cache.py, (4) position cache `remove()` method. Prevents stale markers from blocking future exits on same-symbol re-entries (root cause: Trade 1 success left marker, blocking Trade 2 exit indefinitely). **Exchange Rejection Recovery**: When Schwab API accepts an order (200+order_id) but the exchange rejects it asynchronously, both `_background_verify_order()` (schwab_broker.py) and the order chaser (unfilled_order_chaser.py) now clear `_exit_executed_keys` and call `cache.record_exit_failure(pos_key, reason, is_stop_loss=True)` — enabling market order escalation on retry. Previously, only the "All brokers failed" worker path cleared the dedup key, causing exchange-rejected STC orders to be silently blocked indefinitely.
- **Risk Engine Speed Optimizations**: Interval floor lowered from 1.0s to 0.2s (configurable 0.2–60s range); early-wake sleep chunk 0.1s; `_check_service_enabled()` cached with 10s TTL; `cache.save()` throttled to every 2s. Closing flag uses wall-clock time (`closing_since` field via `time.monotonic()`) instead of cycle count, making timeout independent of loop speed. **Parallel Broker Fetches**: `_fetch_all_positions()` runs ALL broker fetch tasks concurrently via `asyncio.gather` — total cycle time = max(single_broker) instead of sum(all_brokers). Each broker fetch is wrapped in an async closure with its own cache/rate_manager logic. **REST Position Cache (10s TTL)**: Per-broker REST cache reuses results within 10 seconds. Force-bypass via `_force_rest_refresh` flag set on early-wake order events. **SchwabDataHub Hub-First**: Schwab positions fetched directly from `SchwabDataHub.get_positions(detailed=True)` before falling back to REST — zero API cost when hub has data. **Cross-Broker Price Updates**: `_update_prices_from_hub()` updates non-streaming broker positions (Alpaca, IBKR, Tastytrade, Robinhood) with real-time prices from Webull/Schwab streaming hubs for stock symbols — ensures price freshness between REST cache refreshes. **Staleness Guard**: `_get_fresh_hub_price()` enforces a 30s max age (`_HUB_PRICE_MAX_AGE`) on streaming quotes before overriding REST position prices — prevents stale MQTT quotes from low-volume symbols (e.g., TLYS) from overwriting fresher REST data for up to 120s. Cycle timing logged as `[RISK] ⏱ Cycle #N: Xms` (first 5, every 60th, and always when >2000ms). **Stuck Price REST Refresh Fix**: RiskManager accepts explicit `webull_broker` parameter (passed from selfbot as `webull_broker=self.broker`), stored as both `self.webull_broker` and `self._webull_broker`. `_get_raw_webull_client()` checks `_client` before `wb` (selfbot's WebullBroker uses `_client`). New async `_detect_and_fix_stuck_prices()` method runs after `_update_prices_from_hub()` each cycle: tracks per-symbol price change timestamps via `_stuck_price_tracker` dict; when price unchanged >3s (`_STUCK_PRICE_THRESHOLD`), first tries cross-hub price from the OTHER streaming broker (e.g., Schwab hub for Webull positions, Webull hub for Schwab/Alpaca positions) with 5s max age — zero API cost instant fix. If cross-hub has no fresh quote, cascading REST fallback: (1) Webull raw client REST (universal market data, bypasses hub), (2) Schwab REST direct API call (`/marketdata/v1/quotes`, bypasses Schwab hub cache), (3) position's native broker `get_quote()` (Alpaca, IBKR, Tastytrade, Robinhood, Trading212). Covers ALL brokers and works even without Webull connected. REST refresh rate-limited by session: 3s regular hours (9:30a-4p ET), 10s extended hours (4a-9:30a, 4p-8p), 30s closed (nights/weekends) via `_get_market_session()`. All 3 REST sources (Webull, Schwab, broker `get_quote()`) wrapped in `asyncio.wait_for(timeout=5.0)` to prevent rate-limit/429 blocking from stalling risk cycles. Per-cycle budget cap: max 3 REST repairs per cycle (`_MAX_REST_REPAIRS_PER_CYCLE`) — cross-hub (free) checks unlimited, REST-gated. REST cascade now checks `abs(price - current) > 0.0001` per source — skips sources returning the same stale price and continues to next. **REST Repair Guard**: When stuck-fix writes a fresh REST price, `_rest_repaired_prices[key]` stores `{price, until: now+_HUB_PRICE_MAX_AGE(30s)}` — `_update_prices_from_hub()` checks `_is_rest_repair_active(pos, hub_quote_age)` and only overwrites if (a) hub price differs by `max($0.02, 0.3%)` from repaired price (scale-adaptive threshold), OR (b) hub quote age < 3s (stream genuinely recovered — guard released early). Repair keys use `f"{broker}_{symbol}_{asset}"` format consistently across stuck-tracker, repair guard, and hub overlay. Closed positions cleaned from both `_stuck_price_tracker` and `_rest_repaired_prices` each cycle. **Stop-Loss Confirmation After Repair**: `_rest_repair_cycle_keys` map (cleared every cycle start) tracks keys repaired THIS cycle — `_evaluate_all_exit_conditions` early-returns `no_exit()` after pure checks (price-based/channel stops), skipping ALL state-mutating evaluators (tiers, enhanced risk, early trailing, trailing stop, global risk) entirely on repair cycles. This prevents both false exits AND false state mutations (tier escalation, trailing activation, dynamic SL updates) from transient REST prices. Uses map instead of object flag to avoid snapshot-reuse leakage. **Budget Fairness**: Stuck candidates collected first, then sorted by longest stuck duration (descending) before processing — most critical positions get REST budget priority. `tracker['rest_refreshed']` only stamped when REST actually fires (inside budget gate). **Halted Stock Loop Prevention**: Removed `elif fresh_price: tracker['last_changed'] = now` branch — halted stocks stay in stuck state with natural cooldown backoff instead of infinite re-probe loops. Cross-hub stuck-price max_age tightened from 5s to 2s. **Timezone Robustness**: `_get_market_session()` fallback calls `time.tzset()` after setting `TZ=America/New_York` to ensure process respects the change. **Duplicate Detector Removed**: Old 8s `_price_change_tracker` / `_force_webull_rest_refresh` detector inside `_update_prices_from_hub()` removed — single unified 3s detector in `_detect_and_fix_stuck_prices()` with cascading REST chain handles all brokers. Warn throttle changed from broken count-based to time-based (1 msg/60s with total count). REST fallback `_fetch_webull_rest()` uses centralized `_get_raw_webull_client()` with throttled warnings to eliminate log spam.
- **3-Layer Webull Stock Price Fix (Pre-Market SL Protection)**: Root cause: Webull stock positions derive `current_price = marketValue/quantity`, but during pre-market `marketValue` reflects previous day's close, causing false stop-loss triggers (e.g., SER trade: $1.28 prev close shown instead of $1.95 fill → immediate SL exit). **Layer 1** (`_resolve_webull_stock_price()`): Replaces `marketValue/quantity` with prioritized chain: `latestPrice` → `lastPrice` → streaming hub quote (60s max age) → `marketValue` with >25% deviation guard during extended hours. **Layer 2** (`_check_price_freshness()`): First-cycle guard in `_evaluate_exit_conditions` — when price deviates >1.5× SL% from entry during extended hours, cross-checks streaming hub price. If stream shows loss < SL%, substitutes stream price and blocks exit. If stream confirms loss ≥ SL%, allows SL evaluation immediately. If no stream available, defers exactly once per position via `_price_deferred_once` dict, then allows normal evaluation on next cycle. **Layer 3** (enhanced `_detect_and_fix_stuck_prices()`): Previous-close detection triggers forced REST refresh after 2s when price is >15% below entry during extended hours, even if normal stuck threshold not met. Both `_price_confirmed_fresh` and `_price_deferred_once` dicts are pruned of closed positions every cycle alongside `_stuck_price_tracker` cleanup.

- **Broker Isolation**: Prevents data cross-contamination by isolating each broker's streaming data.

- **Hub-First Architecture**: Prioritizes cached streaming data over REST API calls.
